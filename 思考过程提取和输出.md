# 思考过程提取和输出

> 本文说明 **Omics Agent 在 `/api/chat`（及同构编排路径）** 下，从 LLM 到 SSE 再到前端的**思考过程**全链路。目标：为在其他「路由/模式」下一键复现提供**可执行的设计约束与文件索引**。

---

## 1. 目标与数据契约


| 概念       | 约定                                                                                                             |
| -------- | -------------------------------------------------------------------------------------------------------------- |
| **思考**   | 模型内部推理：可能以 `delta.reasoning_content` 或 `content` 内 `<think>…</think>` 出现。                                      |
| **正文**   | 应对用户可见的 `delta.content`（在标签外或 `reasoning_content` 流之外）。                                                        |
| **绝对隔离** | 思考**不得**进入 `state_snapshot["text"]`；正文**不得**进入 `state_snapshot["reasoning"]`（见 `AgentOrchestrator._emit_sse`）。 |
| **时光机**  | 入库/历史恢复依赖 `state_snapshot`：`reasoning` 与 `text` 分轨，与前端 `createRestoreAIMessageRowWithSnapshot` 等四联一致。          |


参考实现：`gibh_agent/core/orchestrator.py` 中 `_emit_sse`（`thought` → `reasoning`，`message` → `text`）。

## 2. 后端分层（从上到下）

### 2.1 厂商 API：是否产出「推理流」

- **GLM（`glm-5.1`）**、**DeepSeek（`deepseek-v4-pro`）**、**Kimi（`kimi-k2.6`）**：在兼容 OpenAI SDK 的场景下，通过 `**extra_body.thinking`** 开启/关闭深度思考（标准/旗舰对应 `disabled` / `enabled`）。
- 实现位置：`**gibh_agent/core/llm_client.py`** → `_inject_vendor_thinking_extra_body`；集合 `_THINKING_TOGGLE_MODEL_IDS`。
- 请求级模式：**ContextVar** `set_llm_api_mode` / `reset_llm_api_mode`（由 `server.py` 的 SSE `generate_sse` 在调用编排器前设置，来源于 `ChatRequest.mode` 归一化）。

**复现要点**：新路由若自行创建 `LLMClient` 并调用 `achat`/`astream`，须在**同一异步上下文**内设置 `llm_api_mode`，否则旗舰/标准不会传到 `extra_body`。

### 2.2 原始流分拣：结构化字段优先，标签兜底

核心模块：`**gibh_agent/core/stream_utils.py`**

- `**stream_from_llm_chunks(chunk_iter, model_name)`**  
  - 若 `**delta.reasoning_content**` 非空 → 产出 `("thought", {"content": reasoning})`。  
  - 否则对 `**delta.content**` 做缓冲区 + `**in_think_tag**`：  
    - 位于 `<think>` / `</think>`（常量 `THINK_OPEN` / `THINK_CLOSE`）之间 → `thought`；  
    - 其余 → `message`。
  - 保留尾部若干字符防跨 chunk 切断标签（`THINK_OPEN_LEN` / `THINK_CLOSE_LEN`）。

**复现要点**：任何「把 `AsyncIterator[ChatCompletionChunk]` 转成对话事件」的路径，应**复用或等价实现**上述双轨逻辑，避免把整段 `content` 只当正文。

### 2.3 DeepReAct：直连 delta

`**gibh_agent/core/react_runner.py`** 中 `DeepReActRunner`：对 `astream` 逐 chunk 读取 `reasoning_content` → `emit_callback("thought", …)`，`content` → `emit_callback("message", …)`。  
与 `stream_from_llm_chunks` 语义对齐，只是接入点不同。

### 2.4 编排器：统一打成 SSE + 写入快照

- `**AgentOrchestrator._stream_chat_mode`**（及同类分支）消费 `stream_from_llm_chunks` 产出的事件，经 `_emit_sse(..., "thought"|"message", data)` 下发。
- `**_emit_sse**`：  
  - `event_type == "thought"` → `state_snapshot["reasoning"] += data["content"]`  
  - `event_type == "message"` → `state_snapshot["text"] += data["content"]`

**复现要点**：新增路由若绕过 `_emit_sse` 自行 `yield` SSE，必须**同步维护**同样的 `state_snapshot` 字段，否则历史/时光机断裂。

## 3. 前端（`services/nginx/html/index.html`）SSE 消费

### 3.1 事件入口

- `**event: thought`**：累加 `thoughtBuffer`；旗舰模式下写入执行记录 `appendReasoningToActiveLog`、可选 `appendReasoningToTextZone`；并可挂 `ensureThoughtBlock`。
- `**event: message`**：正文增量经 `consumeThinkTaggedDelta` 拆分 `thoughtDelta` / `answerDelta`（处理嵌在正文里的 `<think>` 块）；`answerDelta` → `updateChatBubble`。
- `**mode`（`standard` / `flagship`）**：前端用 `sseApiMode`；**标准模式**下可跳过向执行记录/thought 通道写入（避免幽灵「思考」步骤），详见当前 `handleServerEvent` 分支。

### 3.2 流式标签二次分流（防漏网之鱼）

`**consumeThinkTaggedDelta(deltaText)`**（与后端 `stream_utils` 呼应）：

- 维护 `reasoningStreamState.inThinking` + `pendingTag`（防半拉子标签）。
- **字面误伤防护**：若在**正文语境**出现 `<think>` 且当前缓冲内**无闭合**，且片段像「标签/内的内容」等**转述**，则将尖括号 **HTML 转义**写入正文，**不**切入 thinking（避免正文被整体吞进「推理」区）。

### 3.3 正文渲染前清洗（禁止误删用户可见字）

`**updateChatBubble`**：仅做**非贪婪**成对剥离 `<think>…</think>` 等；若全串只有开放标签无闭合，则**转义** `<think>`，**禁止**「从开放标签删到文末」的旧策略。

`**stripThinkForReport`**（右侧报告）：同样只剥夺对闭合块；未闭合时转义而非截断。

### 3.4 持久化到 DOM / 会话

- 流结束：`**persistReasoningToMessageObject`**：`thoughtBuffer` → 当前 AI 行 `dataset.reasoningContent` + `.reasoning-zone` 内 `<details class="thought-process-persistent">`。
- 历史：`**createRestoreAIMessageRowWithSnapshot**` 使用 `snap.reasoning` 填充 reasoning 区（与 SSE 聚合规则一致）。

## 4. 在其他路由模式下「一键复现」的检查清单

1. **API**
  - 是否走 `MODEL_ROUTING_TABLE` 登记的模型？  
  - 是否在本次请求链路中调用 `set_llm_api_mode`（若该模型支持 `extra_body.thinking`）？
2. **流解析**
  - LLM 流是否经过 `stream_from_llm_chunks`（或等价逻辑）？  
  - 是否区分 `reasoning_content` 与 `content`？  
  - `<think>` 是否只在成对闭合时从正文剥离？
3. **SSE 形状**
  - 是否下发 `event: thought` / `event: message`（或前端已适配的同名事件）？  
  - 快照是否通过 `_emit_sse` 写入 `reasoning` / `text`？
4. **前端**
  - 是否处理 `thought` 且不与 `message` 混写进同一展示字段？  
  - Markdown 入口是否避免「未闭合标签截断全文」？  
  - 是否需要 `sseApiMode` 同类开关（标准/旗舰）？
5. **文档/宪法**
  - 变更 SSE 或 `state_snapshot` 时，对照 `docs/思考过程和历史快照.md`（若存在）或仓库规则中的 **时光机四联**。

## 6. 已知误区（刻意避免）

1. ~~「右侧核弹：未闭合 `<think>` 起删到文末」~~ → 已改为**非贪婪成对删除 + 未闭合转义**，否则会误删正文中口述「`<think>`」的后续文字。
2. **不要把思考写入 `text` 快照**：入库后历史气泡会把推理当正文重复展示。
3. **累计型 `workspace_update.markdown`**：必须用 raw/clean 双缓冲算 **delta**，再送进 `consumeThinkTaggedDelta`，否则会重复解析或覆盖 thought。

## 7. 与旧版文档的差异说明

- 早期草稿中「去除未闭合起始标签后的全部内容」**已不再作为推荐策略**；当前实现以 **成对匹配 + 字面转义** 为主，并与前端 `consumeThinkTaggedDelta` 的启发式一致。  
- 「思考只进左侧执行记录」已扩展为：**执行记录条目的 `reasoning_content` + reasoning 折叠区 + `state_snapshot.reasoning`** 多轨一致。
