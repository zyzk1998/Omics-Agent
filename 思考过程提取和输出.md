# 思考过程提取和输出

## 目标

- 思考流（`reasoning_content` 或 `<think>...</think>`）只进入左侧执行记录。
- 右侧正式报告只显示净化后的正文，绝不显示思考过程。

## 一、推荐的数据流分层

- **思考流通道**：进入 `thoughtBuffer` 和执行记录状态（如 `activeExecutionLogs[].reasoning_content`）。
- **正文通道**：进入 `reportBuffer`（或 `workspaceCleanBuffer`），仅用于右侧 Markdown 渲染。
- **禁止混流**：不要让原始模型输出直接进入右侧渲染。

## 二、流式分片处理模式（关键）

### 1) 优先使用结构化字段

- 若 SSE payload 有 `reasoning_content`，优先视为思考流，直接写入左侧状态。
- 若无结构化字段，再处理 `<think>` 标签分流。

### 2) 使用“增量 delta”而不是全量覆盖

- 许多 `workspace_update.markdown` 是“累计全文”，不是增量。
- 必须维护：
  - `workspaceRawBuffer`：上一次原始全文
  - `workspaceCleanBuffer`：已净化正文
- 当前增量 `delta` 计算规则：
  - 若 `current.startsWith(prevRaw)`，则 `delta = current.slice(prevRaw.length)`
  - 否则视为重置/异常，按全量处理

### 3) 状态机分流

- 对 `delta` 进行状态机解析：
  - `thoughtDelta` -> 左侧状态
  - `answerDelta` -> `workspaceCleanBuffer`
- 严禁把 `thoughtDelta` 写入右侧 buffer。

## 三、右侧“核弹级兜底”策略

在右侧 Markdown 最终入口（`safeMarkedParse` 之前）强制执行：

1. 去除完整思考块：
   - `/<(?:think|redacted_reasoning)>[\s\S]*?<\/(?:think|redacted_reasoning)>/gi`
2. 去除未闭合起始标签后的全部内容：
   - `/<(?:think|redacted_reasoning)>[\s\S]*$/gi`

这保证了即使分片中只收到 `<think>` 没有 `</think>`，右侧也不会泄漏任何思考内容。

## 四、左侧 UI 绑定最佳实践

- 使用响应式状态（数组/对象）驱动渲染，不依赖临时 DOM 强插。
- 每条执行记录包含：
  - `content`
  - `state`
  - `reasoning_content`
- 渲染模板建议：

```html
<details>
  <summary>思考过程</summary>
  <pre>{{ reasoning_content }}</pre>
</details>
```

## 五、排查清单

- 左侧无思考：
  - 检查是否处理了 `reasoning_content` 字段。
  - 检查是否对“累计全文”误当增量导致 thought 被覆盖。
- 右侧泄漏思考：
  - 检查最终渲染入口是否执行了兜底清洗。
  - 检查是否有旁路渲染路径绕过清洗。
- 报告截断：
  - 检查 SSE 结束时 `TextDecoder` 尾部 flush。
  - 检查是否正确处理 `[DONE]` 与残余 buffer。
