# Weekly Report Writer (周报撰写助手)

## 概览

此技能自动化了周报/双周报的创建过程。它通过查找指定日期范围内的所有相关文档，阅读其内容，并根据模板或分层结构合成信息。它特别注重项目的连续性（追踪上一份报告的任务状态）和内容的深度（通过链接溯源）。

## 何时使用

当用户要求：

- 「写我的周报」或「起草本周摘要」。
- 总结两个日期之间的进度。
- 基于上一份报告和一组新文档创建新报告。

此技能在文档知识库（如 Obsidian 知识库）中特别有效。

## 工作流

遵循此工作流生成报告。

### 第一步：收集用户输入（70 分边界 — 勿硬编码阻断）

自评信息完整度：**<40%** 才追问（最多 2 个核心问题）；**≥70%** 必须直接成稿，次要缺口用 `[待补充：…]` 占位。**无任何输入**时输出高质量 **Demo 周报**。

理想情况下可收集（非全部必需）：

- **开始日期**：报告周期的开始。
- **结束日期**：报告周期的结束。
- **仓库文件名是否包含日期前缀**：确认文件命名约定（如 `YYYY-MM-DD-描述.md`）。
- **上一份报告路径**：上一份周报的文件路径。这对追踪正在进行的任务至关重要。
- **笔记库目录**：文档存储库的根目录（例如 Obsidian vault 路径）。
- （可选）**报告模板路径**：如果用户有特定模板，请提供其路径。
- （可选）**项目背景路径**：总结更大组织所有项目的文档路径（如果有）。

### 第二步：查找文档

使用用户提供的 **笔记库目录** 和日期范围，查找日期范围匹配的所有 Markdown 文件。

### 第三步：与用户确认文件列表

向用户展示找到的文件列表（日志 + 探测到的模板）。请用户确认列表，并反馈任何应 **重点关注** 或 **跳过** 的文件。

- 摘抄的文档、语音转写原文日志等应 **跳过**。
- 每日工作小结、会议纪要、项目更新等应 **重点关注**。
- 个人 Idea 或 side project 要记录在第一部分的末尾一节。

### 第四步：阅读、深度溯源与合成

确认后，阅读所有文件的内容。过程中要注意：

**链接追踪**：在阅读每日摘要/日志时，若遇到指向另一文档的「关键事项」且包含 `[[文档链接]]`，应读取被链接文档。每日摘要通常只是一句话，高质量周报需要源文档中的具体产出、方法和数据。

**信息合成**：

- 追踪「上一份报告」中的项目，凸显其进展。
- 识别待办 🟥、已完成 🟩、进行中 🟨 和新想法 💡。
- **待办继承**：上一份报告中任何 🟥 或 🟨 的项目，必须带入新报告的「待办/积压」区域，除非已明确标记为 🟩。

### 第五步：起草报告（分层输出）

若找到特定模板，严格遵循；否则使用标准分层结构：

**第一部分：个人状态同步**

- 受众：自己 / 知识库。
- 内容：详细技术背景、指向深度文档的链接、个人思考 💡 和具体状态更新。保留 `[[Wiki Link]]` 格式。

**第二部分：团队对外同步**

- 受众：管理者 / 团队。
- 内容：高层级摘要；关键进展（交付与价值）；风险/卡点；下一步计划；使用 `@用户` 标记负责人。

报告文件名与文档库既有风格保持一致。

### 第六步：遗漏检查 (Gap Analysis)

草稿生成后做遗漏检查，提醒用户「检查草稿，若有遗漏请补充」，并按反馈修订。

## 输出格式

完整 Markdown 报告，必须包含清晰的「第一部分」「第二部分」标题；状态标记使用 🟥🟩🟨💡。

**排版铁律（商务紧凑）**：
- 文首 `>` 引用块：本周一句话摘要。
- `### 本周高光` / `### 核心问题` / `### 下周计划` 三节。
- 进度用 `- [x]` / `- [ ]` 任务列表；缺失数据写 `[待补充：具体数据/核心成果]` 等占位符，勿留空节。
