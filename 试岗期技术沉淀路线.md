# 试岗期技术沉淀路线

> 用途：转正/试岗述职 PPT 的底层素材；叙述以可核验代码与公开仓库说明为准，不展开修辞。  
> 对照仓：**当前成熟态** = 本仓库 `GIBH-AGENT-V2`；**上一代容器化产品** = 宿主机并列目录 `~/GIBH-AGENT`；**更早期底座实验** = 公开仓 [agent-mini](https://github.com/zyzk1998/agent-mini)（curl/Ollama/ReAct 闭环）、[gpt-oss-web](https://github.com/zyzk1998/gpt-oss-web)（若述职需写 Web/网关形态可与之对齐；本工作区未检出其目录）。

---

## 一、原点与混沌期（探索阶段）

### 1.1 agent-mini：验证「推理–行动–观测」闭环

- **要解决的核心问题**：在**不引入重型编排框架**的前提下，证明本地模型可通过 HTTP 完成 **OpenAI 风格 `tools` 调用 → 真执行 → 结果回灌 → 再对话** 的多轮闭环（公开 README 中的 6 步流水线：`tools.json` + `curl` + `jq` + bash）。
- **技术取舍**：Ollama REST、`stream: false`、**零 LangChain**；显式记录「哪些镜像支持 tools、哪些不支持」——属于**可行性验证与接口心智**沉淀，而非产线架构。
- **瓶颈**：
  - **无会话状态/无资产中台**：无上传文件谱系、无权限与审计链。
  - **无并发与隔离**：无 API 层队列、无计算与推理进程隔离；长任务会占满同一调用路径。
  - **工具形态**：手写 JSON Schema 片段，与后续「从代码签名自动生成 Schema」未对齐，维护成本高。

### 1.2 GIBH-AGENT（`~/GIBH-AGENT`）：单细胞场景第一次「产品化」编排

- **要解决的核心问题**：把能力从脚本抬到 **Docker Compose 常驻服务**：Nginx 入口、FastAPI API、**Celery + Redis** 异步 worker、**双 vLLM**（逻辑模型 + 多模态 VL）OpenAI 兼容端点；`services/api/src/skills/` 下 Scanpy 等技能以 **Python 文件 + `META` 字典** 被 `SkillManager` 扫描加载（`importlib` + `glob`）。
- **已解决的痛点（相对 agent-mini）**：
  - **长耗时分析不阻塞 HTTP**：分析进 Celery；推理与业务进程分离。
  - **双脑**：逻辑链路与视觉链路分到不同 vLLM 服务，缓解单容器「又聊天又跑重推理」的争用。
- **仍存在的瓶颈（混沌残留）**：
  - **意图与步骤强耦合关键词**：`BioBlendAgent.process_query` 用 `action_keywords` / `bio_keywords` 等子串触发规划；**扩展模态或模糊说法易误判**，且难以审计「为何判成规划」。
  - **工作流步骤表硬编码**：`FULL_STEPS_REGISTRY` 以静态列表描述标准 scRNA 步骤；新增步骤需改核心文件，**非数据驱动的工具注册表**。
  - **技能匹配偏薄**：`SkillManager.match_skill` 以 `id`/`name` 子串出现在 query 中为命中条件，**无结构化路由输出**，易与真实用户自然语言漂移。
  - **幻觉与工具边界**：规划路径依赖 LLM JSON，虽有 `_repair_json` 等修补，但**缺少全局「chat vs task vs 澄清」分层护栏**与「动态工具视野」约束模型勿编造工具名。
  - **栈依赖**：LangChain 封装 ChatOpenAI 指向 vLLM；版本与抽象层带来排障成本（与后期「编排器直驱 LLMClient」方向不一致）。

## 二、破局与重构期（架构萌芽）

> 时间线以「技术决策」为单位，不强行对应日历月份；述职时可自行钉到季度。

### 2.1 从「关键词意图」到「分层路由 + 可校验结构化输出」

- **动机**：关键词与硬编码 registry 无法覆盖多模态组学 + 化学 + 超算运维 + 纯检索的交叉表述；需要**一层轻量、可观测、可降级**的全局判决。
- **落地（当前仓）**：
  - `gibh_agent/core/semantic_router.py`：**独立 LLM 调用**输出 `route`（`task` / `hpc` / `chat` / `clarify` / `skill_fast_lane`）+ `confidence` + `rationale_short`；**Pydantic `RouterOutput` 强校验**；置信度低于阈值 **最多重试 2 次**，失败统一 **clarify 兜底**。
  - **静态红线（A–E）** 写在 `SEMANTIC_ROUTER_SYSTEM_PROMPT_BASE`：无数据禁 task、无 MCP 开关禁 hpc、含生物学术语但无执行诉求走 chat、化学成药单点执行走 skill_fast_lane 等。
  - **动态护栏（F 段）**：`build_semantic_router_system_prompt()` 注入 `build_capabilities_digest_for_llm()`——从 **WorkflowRegistry** 与 **ToolRegistry** 拉真实 SOP 名与 `tool_id` 节选，减少「编造不存在的工具/流程」。

### 2.2 从「目录扫描 META」到「单例注册表 + 反射式参数 Schema」

- **动机**：工具数量上升后，手写参数表与前端/执行器三方易漂移；需要 **OpenAI `tools[].function.parameters` 与入库/RAG 同源**。
- **落地（当前仓）**：
  - `gibh_agent/core/tool_registry.py`：`@registry.register` 装饰器从函数签名 **`inspect` + `typing` 生成 Pydantic `args_schema`**，`get_all_tools_json()` / `openai_tools.tool_names_to_openai_tools` 使用 **`model_json_schema()`**。
  - `gibh_agent/tools/__init__.py`：`pkgutil.walk_packages` **递归 import** 触发注册，保证新增模块即注册（即插即用语义：**包内发现即生效**，而非运行时 DLL 热插拔）。

### 2.3 从「Celery 包一切」到「控制平面 / 数据平面 + 重型算子外移」

- **动机**：主 API 进程内跑 RDKit/长推理会相互拖死；需与架构宪法一致：**重 CPU/GPU 外移**（独立 Worker / 子进程 / TaaS）。
- **落地（当前仓，摘 docker-compose 与代码惯例）**：
  - **MySQL**：用户、技能广场、动态插件等持久化（相对 GIBH-AGENT 以文件+Redis 为主的形态，**中台化**）。
  - **worker-pyskills**、**`CHEM_RDKIT_PYTHON` / `run_chem_tool.py`**：化学原子工具子进程隔离。
  - **MCP 网关**：`hpc_mcp_*` 工具动态注册/注销（`ToolRegistry.unregister_many`），与 **compute_scheduler** 开关联动（`openai_tools.apply_hpc_mcp_tool_policy`），避免无超算时模型侧「空挂」HPC 工具。

### 2.4 编排器层的「窄短路」与资产嗅探（与路由互补）

- **动机**：避免「点工作流确认」或 `target_domain` 等**强执行信号**再被语义路由抖动；避免 PDB/文档类上传被误送进默认 RNA DAG。
- **落地**：`gibh_agent/core/orchestrator.py` 中 `_forced_task_fast_lane`、`_is_public_api_chat_intent`、**OmicsAssetManager 嗅探** + 模糊分析查询走 chat 等（与 Semantic Router 形成 **规则快车道 + LLM 路由** 双轨）。

## 三、工业化成型期（当前态 / `GIBH-AGENT-V2`）

### 3.1 原子化工具库与注册机制

- **单例 `ToolRegistry`**：统一 `name / description / category / output_type / args_schema`；支持 **别名**与 **MCP 工具批量注销**。
- **注册触发链**：包导入 → `load_all_tools()` 遍历子模块 → `@registry.register` 写入 `_tools` 与 `_executables`。
- **对外契约**：与 **OpenAI Function Calling** 对齐的 **JSON Schema（Pydantic 生成）**；另有 `scripts/export_tool_library_md.py` 导出 `工具库.md` 便于评审与 prompt 对齐。

### 3.2 结构化路由与 Pydantic 校验

- **输入**：`RouterInput`（`query`、`file_status`、`mcp_status`、`session_flags`、`recent_history`）。
- **输出**：`RouterOutput` 三键硬契约；解析走 `extract_json_object_from_llm_text`，防止 Markdown 围栏破坏 JSON。
- **阈值策略**：`CONFIDENCE_THRESHOLD = 0.7`，失败路径可观测（日志 + SSE「路由判定分析」文案）。

### 3.3 静态红线 / 动态护栏

- **静态**：Prompt 内 A–E 段（数据前提、负向护栏、skill_fast_lane 触发语义、hpc 与 MCP 绑定、跨轮历史）。
- **动态**：F 段 **capabilities digest**（SOP + 按类别节选的真实 `tool_id`）；编排器内 **公开库轻查询**、**社交礼仪极窄短路** 等与路由并列。

### 3.4 快车道（Fast Lane）

- **语义层**：`RouteKind.skill_fast_lane`（化学成药/相似性等 **PySkills 单点执行** 与纯概念 chat 分离）。
- **编排层二次消歧**：在 `drug_similarity` 语境下，用用户原文关键词在 **`lipinski_druglikeness`** 与 **`drug_similarity_search`** 间选择，并 **补写 `[Skill_Route: …]`** 再交 `SkillAgent`（避免仅靠模型口头承诺工具名）。

### 3.5 其它「成熟态」要点（PPT 可一笔带过）

- **多模态资产总线（OmicsAssetManager）**、**执行器反射注入与签名驱动参数**、**SSE `state_snapshot` 与前端历史恢复四联**（见项目 `.cursorrules` / `docs/技能扩展规范文档.md` 等）；此处不展开，避免与「本文件主线：路由–工具–算力分层」抢篇幅。

## 四、演进总结

**一句话**：从 **bash/curl 证明工具闭环**（[agent-mini](https://github.com/zyzk1998/agent-mini)）→ **Docker + Celery + 双 vLLM 的单细胞产品骨架**（`~/GIBH-AGENT`，关键词 + 硬编码步骤表）→ **可校验结构化全局路由、Pydantic 驱动原子工具注册表、静态红线与动态能力 digest、化学/HPC 快车道与重算力外移** 的 **Omics-Agent / GIBH-AGENT-V2 工业编排体**。
