📋 全流程数据流向详解 (V2 Architecture)

本文档与上方交互式场景模式（Planning / Execution / Error Loop / Direct Chat）一一对应，采用白盒视角描述系统内部机制。

一、核心机制解析 (Core Mechanisms)

1. 文件治理与数据感知 (File Governance & Data Perception)

  • Structure Normalizer：自动解压 .zip / .tar.gz 并重组目录（如配对 .h5 与 spatial/），下游接收解压后路径。
  • Modality Sniffer：按文件内容与结构自动检测模态（Spatial / Radiomics / RNA / Metabolomics 等），支持 HITL 在未知时由用户选择。
  • File Inspector：多模态文件检查器（策略模式），如 SpatialVisiumHandler、RadiomicsHandler 等，支持 10x Visium、AnnData、NIfTI、CSV/TSV、FASTQ 等，输出 FileMetadata。
  • Logic Branch：根据 has_file / dominant_modality 触发 Preview 或 Execution 模式。

2. 动态规划 (Dynamic Planning)

  • CoT + RAG：Planner 结合 Chain-of-Thought 与 RAG（Tool Retriever 语义检索），生成工作流。
  • Step 勾选/取消：根据用户意图（如「跳过预处理」「不要通路富集」「只提特征」）解析 target_steps 与 skip_steps，对应步骤 selected: false，实现 Dynamic Planning。
  • Param Tuner：基于 FileMetadata 与 Data Shape 动态推荐参数（如 LOESS/UMAP、Leiden 分辨率等）。

3. 闭环反馈与领域专属摘要 (Closed-Loop & Domain-Specific Summary)

  • Result Rollback：_read_execution_results(output_dir) 读取摘要与图片路径，注入 LLM 上下文。
  • Domain-Specific Summary：各领域 Agent（Radiomics、Spatial 等）重写 _generate_analysis_summary()，使用本领域专属 Prompt，避免出现代谢组学等跨领域话术。
  • Error Self-Healing：Executor 将 error_context 反馈 Planner，LLM 生成 corrected_params 后重试。

二、场景化流程拆解 (Scenario Walkthroughs)

Phase 1: 文件治理与规划推理 (File Governance & Planning)

对应场景按钮：规划流程 (Planning Flow)

  • User → 发送查询与文件（含压缩包/医学影像）→ Orchestrator
  • File Governance → StructureNormalizer 解压归档 → ModalitySniffer 检测模态 → FileInspector（SpatialVisiumHandler / RadiomicsHandler 等）→ effective_root, dominant_modality, FileMetadata
  • Orchestrator → 规则检查 + FileMetadata → 决定 Preview / Execution 模式
  • Planner → Dynamic Planning（解析 skip_steps / target_steps）→ Retriever（ChromaDB）→ Tool Schemas
  • LLM → CoT 推理 → JSON Workflow（含 selected）→ 前端渲染工作流卡片

Phase 2: 执行与自愈 (Execution & Self-Healing)

对应场景按钮：执行流程 (Execution Flow) / 错误循环 (Error Loop)

  • Agent → 按模态路由到领域智能体（RNA / Metabolomics / Spatial / Radiomics 等）→ Executor
  • Executor → ToolRegistry 查找工具 → Pydantic 验证 → 调用工具（含步骤间 h5ad_path 等链式传递）
  • Tools → 执行分析 → 返回 step_result；前端对非图片结果使用 Universal File Card 渲染
  • Error Loop：Executor 上报错误 → Planner + LLM 修正方案 → Executor 重试
  • Result Aggregation → steps_details, steps_results, output_dir

Phase 3: 领域专属报告生成 (Domain-Specific Report)

对应场景：执行完成后的 diagnosis 事件

  • Execution Summary → steps_results + output_dir
  • Domain-Specific Summary → 各 Agent 使用本领域 _generate_analysis_summary()（如 Radiomics / Spatial 专属 Prompt），非通用代谢组学话术
  • SSE Stream → diagnosis 事件 → 前端渲染 Markdown 报告与 Universal File Card / 图片

三、技术栈映射 (Tech Stack Mapping)

  • Modalities：RNA、代谢组学、空间组学 (Visium)、影像组学 (NIfTI) 等多模态领域 Agent
  • Reasoning：DeepSeek-R1 (MoE) — 意图分类、Dynamic Planning、错误分析、领域专属诊断报告
  • Memory：ChromaDB — 工具语义检索、RAG
  • Scheduling：Celery + Redis — 异步任务调度
  • Streaming：SSE — status / workflow / step_result / diagnosis；前端 Universal File Card 渲染非图片结果
  • Safety：Pydantic + Docker 沙箱；内生安全：Celery 异步 BLAKE3 + Ed25519 签名、.sig 旁路校验、Fail-Open 设计

🔑 关键技术点

1. 工具自动发现与注册

系统通过 Python 装饰器和反射机制，自动发现和注册工具到 ToolRegistry。新增工具只需创建文件并添加 @registry.register 装饰器，无需修改核心代码。

2. 语义工具检索

使用 ChromaDB 向量数据库和 Ollama Embeddings，实现基于语义相似度的工具推荐。用户可以用自然语言描述需求，系统自动推荐相关工具。

3. 动态规划 (Dynamic Planning)

将工具 Schema 注入 LLM 提示词，根据用户意图（如「跳过预处理」「不要通路富集」）解析 target_steps / skip_steps，动态勾选或取消步骤，生成可定制工作流，支持 Spatial、Radiomics 等模态的 Few-shot 示例。

4. 通用执行引擎与多模态工具

基于策略模式与依赖注入，实现工具无关的执行框架。支持 RNA、代谢组学、空间组学、影像组学等多模态工具（Scanpy、Squidpy、PyRadiomics 等），步骤间数据链式传递（如 h5ad_path）。前端对非图片结果使用 Universal File Card 渲染。

5. SSE 流式输出

使用 Server-Sent Events (SSE) 实时推送分析进度、步骤结果与领域专属 AI 报告，前端解析事件并渲染工作流卡片、步骤结果与 Markdown 报告。

6. 多用户隔离与内生安全

多用户独立目录（{user_id}/{session_id}/）。内生安全：Celery Worker 异步 BLAKE3 哈希 + Ed25519 签名，.sig 旁路文件做完整性校验，Fail-Open 设计保证业务可用。

📋 步骤详解

  1. 前端 → API 服务器：HTTP POST 请求（文件上传含压缩包/医学影像、聊天请求）
  2. 文件治理：StructureNormalizer 解压 → ModalitySniffer 检测模态 → FileInspector 提取元数据
  3. API 服务器 → 编排器：创建 AgentOrchestrator，调用 stream_process()
  4. 编排器 → 智能体/规划器：意图分类、按模态路由到领域智能体（RNA/代谢/空间/影像组学等）、调用规划器（Dynamic Planning）
  5. 规划器 ↔ 工具检索器/LLM：语义检索工具、LLM 生成工作流（含 skip_steps/selected）
  6. 领域智能体 → 执行器：创建 WorkflowExecutor，执行工作流
  7. 执行器 → 工具：动态查找工具、验证参数、步骤间数据链式传递、调用工具函数
  8. 工具 → 执行器：返回 step_result（数据、图表路径）；前端非图片用 Universal File Card 渲染
  9. 执行器 ⇢ 规划器：错误时反馈 Traceback，触发重规划
  10. 领域智能体 → 编排器：领域专属摘要（_generate_analysis_summary）生成报告
  11. 编排器 → 前端：SSE 流式输出（status, workflow, step_result, diagnosis, result, done）
  12. 前端渲染：解析 SSE 事件，渲染工作流卡片、步骤结果、AI 报告
