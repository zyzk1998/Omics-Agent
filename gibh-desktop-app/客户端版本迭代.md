# 客户端版本迭代 — 运维全流程说明

本文说明从 **代码修改** 到 **Windows / Linux 安装包构建**，再到 **前端下载页与自动更新元数据同步** 的典型做法，以及可直接执行的命令（假定仓库根目录为 `/home/ubuntu/GIBH-AGENT-V2`）。

---

## 1. 流程总览（运维视角）


| 阶段              | 做什么                                                                                                                                                                                                                             | 产出 / 落点                                 |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| 改代码             | 编辑 `main.js`、`package.json` 等                                                                                                                                                                                                   | 本地工作区                                   |
| 升版本号            | 修改 `gibh-desktop-app/package.json` 的 `version`（建议同步 `description`）                                                                                                                                                              | 语义化版本如 `0.12.7`                         |
| 配置发布地址（可选但强烈建议） | ① 仓库根 `.env` 中 `PROD_SERVER_URL`（打进安装包的 Web/API 基址）；② `package.json` → `build.publish[0].url` 中 `**YOUR_SERVER_URL`** 替换为公网可访问的站点根（自动更新 generic 源）；③ 或在运行环境配置 `OMICS_AUTO_UPDATE_BASE_URL` / `OMICS_AGENT_WEB_URL`（见 `main.js`） | 客户端连得上网页与更新服务器                          |
| 准备打包环境          | `npm run prepare-pack-env`：从根目录 `.env` 读取 `PROD_SERVER_URL` 写入 `dist-pack.env`（**不会**把完整 `.env` 打入包，避免密钥泄露）                                                                                                                     | `gibh-desktop-app/dist-pack.env`        |
| 构建安装包           | `electron-builder` 打 Windows（NSIS `.exe`）与/或 Linux（AppImage）                                                                                                                                                                    | `gibh-desktop-app/dist-out/`            |
| 同步到 Nginx 静态站   | `npm run sync:download-page`（已包含在各 `dist*` 脚本末尾）                                                                                                                                                                                | 写入并复制到 `services/nginx/html/downloads/` |
| 部署上线            | 将 `downloads/` 中实际文件与 Git 中已跟踪的元数据一同发布到 Nginx 根；重载 Nginx                                                                                                                                                                        | 用户可访问 `/downloads/` 与自动更新 yml           |


**注意**：`services/nginx/html/downloads/` 下部分二进制（`*.exe`、`*.AppImage`、`latest*.yml`、`.blockmap` 等）通常被 `**.gitignore` 排除**，不进入 Git；**生产环境**需用 **rsync / scp / 发布流水线** 把构建产物同步到服务器，与仓库内 `client-release.json`、`index.html` 一致。

## 1.1 近期版本摘要（维护记录）

以下随代码迭代同步更新；详细运维步骤仍以本文其余章节为准。

### V0.13.5

- **Sidecar 生命周期**：`local_server.py` 新增 **`POST /api/shutdown`**（短延迟后 `os._exit(0)`），释放 **8019**。Electron **`will-quit`** 中先 **`curl` 同步调用该接口**，再对本进程 **`spawn` 的 Sidecar** 执行 **`taskkill /PID … /T /F`（Windows）或 `kill -9`（Linux）**，避免退出后 Python Sidecar 僵尸占用端口导致后续会话命中旧路由。
- **移除 openapi 能力不足弹窗**：进程问题改由上述退出清理解决，不再对用户弹「Sidecar 能力不足」。
- **`/api/chat` 载荷**：`index.html` 中 **`local_files`** 仅保留仍属主机本地的路径；已通过 **`upload_to_cloud`** 写入 **`/app/uploads`**（或同源云端 URL）的路径不再写入 **`local_files`**，避免后端按「本地轨」误解析；**`uploaded_files`** 仍为智能体分析用的云端路径列表。
- **小文件 → 智能体闭环**：挂载文件 → **`runLocalAttachmentSmartGate`**（`check_file` → **`upload_to_cloud`**，`Authorization` / **`X-Guest-UUID`** 由 Sidecar 转发至宿主 **`/api/upload`**）→ 更新 **`currentAttachments`/`uploadedFiles`** 为 web 路径 → **`sendMessage`** 组装 **`payload.uploaded_files`**；须保证 **`PROD_SERVER_URL`/`OMICS_AGENT_WEB_URL`** 与网页同源且用户已登录或有访客 UUID。

### V0.13.0

- **本地文件树交互升级**：右侧本地项目栏新增文件搜索框（实时筛选、目录级联保留），提升大型工作区可达性。
- **拖拽注入对话**：文件树文件节点支持拖拽到输入区，按光标位置插入 `[本地文件: 绝对路径]`，并保持输入框自动伸缩。
- **自定义右键菜单**：新增 VS Code 风格菜单项（添加到对话框 / 复制路径 / 预览）；预览通过 `127.0.0.1:8019/api/tools/read_file` 读取文本内容，非文本或读取失败统一提示暂不支持。
- **下载页与元数据**：升版后需同步 `client-release.json`、`downloads/index.html` fallback 版本与安装包文件名，确保前端链接与自动更新元数据一致。

### V0.13.1

- **右侧栏交互深化**：本地项目/超算资产/文件管理统一为右侧可折叠面板，点击当前激活入口可直接收起。
- **侧边栏可调宽**：新增左侧边缘拖拽手柄，支持按视口动态调宽（250px ~ 50%）。
- **实体附件挂载**：文件拖拽与右键“添加到对话框”改为附件卡片，不再写入纯文本；发送消息时附带结构化路径字段。
- **欢迎页排版精修**：英雄栏与 Slogan 位置上移与间距对齐，优化首屏视觉重心。

### V0.13.4

- **启动自检（已在 V0.13.5 调整）**：曾通过 `openapi.json` 校验 `/api/tools/check_file` 并弹窗；现改为 **退出时剿灭 Sidecar 进程**，不再依赖该弹窗。
- **发版要求**：`npm run dist:*` 前必须 `npm run build-sidecar`（`predist*` 已链式），确保 `resources/local_sidecar` 内二进制与 `local_server.py` 同步（**含 `/api/shutdown`**）。
- **本地附件链路其余故障点（排查清单）**：① 8019 仍被占用 → 完全退出客户端或结束残留 `local_sidecar`；② 网页 Toast → 部署最新 `index.html`；③ `upload_to_cloud` 失败 → `PROD_SERVER_URL`、登录态、`/api/upload`；④ 附件须为绝对路径。

### V0.13.3

- **check_file 双通道**：Sidecar 支持 `POST /api/tools/check_file`（JSON `path`）；前端称重优先 POST，遇 404/405 回退 `GET ?path=` + `encodeURIComponent`，兼容仅含 GET 的旧打包 Sidecar。
- **安装包**：Windows/Linux 安装包内置上述 Sidecar 变更；远程网页升级后即使短期未重装客户端，闸门仍可回退工作。

### V0.13.2

- **后端路径三态**：`path_resolver` 区分 Web（相对→`/app/uploads`）/ Local（Windows 盘符与 UNC）/ HPC（前缀可配置）；禁止盘符路径与 `UPLOAD_DIR` 错误拼接；`FileInspector` 格式门禁与存在性校验解耦。
- **Sidecar 增强**：`check_file` 附带 `size_bytes`；`upload_to_cloud` 以 `httpx` multipart 转发至宿主 `/api/upload`；`check_env` / `install_sandbox` 占位接口；依赖增加 `httpx`。
- **前端发送闸**：`runLocalAttachmentSmartGate` 按 200MB 分流——小文件静默上传后将会话附件替换为云端路径；大文件/目录弹出 `#sandbox-modal`，取消终止发送，一键配置后恢复挂起请求（本地绝对路径）。

### V0.12.7

- **Sidecar 打包与路径**：`package.json` → `build.extraResources` 增加 `{ "from": "local_sidecar", "to": "local_sidecar" }`，安装包在 `app.asar` 外携带 Python 侧车目录；`main.js` 用 `app.isPackaged ? process.resourcesPath : __dirname` 解析 `local_server.py` 与工作目录，避免生产环境找不到脚本。
- **Sidecar 可观测性与容错**：`spawn` 后完整采集 `stdout`/`stderr`，监听 `error`/`close`；解释器不存在时按平台回退（如 Windows：`python` → `py -3` → `python3`）；进程异常退出或缺少依赖时用 `**dialog.showErrorBox`** 展示捕获输出（禁止静默失败）。
- **自动更新 UI**：检查更新阶段不再顶栏打扰；仅在检测到新版本或下载/出错时再提示（详见主进程/前端约定）。
- **挂载工作区**：连接 `127.0.0.1:8019` 失败时对用户给出明确提示（本地 Python 依赖与服务状态）。
- **下载页 / 元数据**：升版后务必执行 `**npm run sync:download-page`**（或各 `dist*` 末尾自动执行），使 `services/nginx/html/downloads/client-release.json`、`latest-linux.yml` 与 `package.json` 一致；`**latest.yml`（Windows 自动更新）** 仅在与 **Windows 安装包** 同次构建产物存在时，才应从 `dist-out/latest.yml` 覆盖到站点。若仅执行 `**npm run dist:linux`**，工作区里可能仍残留**旧版** `dist-out/latest.yml`，**切勿**把它复制覆盖线上（否则会与尚未更新的 `.exe` 不一致）；此时应保留线上与当前已部署 **Setup exe** 同版本的 `**latest.yml`**，或在补齐 `**npm run dist:win`**（或 `**dist-win-via-docker.sh**`）后再跑一次 `**npm run sync:download-page**`。

### V0.12.6

- Local Sidecar（8019）、本地工作区与文件树、Electron 生命周期拉起/回收 Sidecar、双轨读写等（详见下载页 Changelog）。

## 2. 发版前检查清单

- `gibh-desktop-app/package.json` 中 `version` 已更新。
- 根目录 `.env` 中 `PROD_SERVER_URL` 指向**目标环境**后端/网页基址（勿把含密钥的整文件复制进客户端资源）。
- `build.publish[0].url` 中占位 `**YOUR_SERVER_URL`** 已替换为真实域名或合规发布地址（参见该文件内嵌 URL 注释说明）。
- 计划同步 `**latest.yml` / `latest-linux.yml`** 及 `***.blockmap`**（脚本会自动从 `dist-out` 复制）；缺失则 **electron-updater** 无法正常工作。
- 按需更新 `services/nginx/html/downloads/index.html` 中的 **Changelog**（纯运维可选步骤）。

cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app

## 4. 可执行命令（按场景）

以下均在 `**gibh-desktop-app**` 目录下执行（先 `cd`）。

### 4.1 仅生成打包用环境片段（一般不单独执行）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run prepare-pack-env
```

### 4.2 仅同步安装包与元数据到前端 downloads（不打新包）

当 `dist-out/` 里已有与 `package.json` 版本一致的产物时：

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run sync:download-page
```

作用概要：写入 `services/nginx/html/downloads/client-release.json`，复制匹配的 `.exe` / `.AppImage`，复制 `latest.yml`、`latest-linux.yml`、`*.blockmap`，复制 `client-app-icon.png`，并清理同名旧版本安装包文件。

### 4.3 仅构建 Linux（x64 AppImage）并同步

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist:linux
```

### 4.4 仅构建 Windows（x64 NSIS）并同步（本机需 Wine；Linux 无 Wine 会失败）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist:win
```

### 4.5 在无 Wine 的 Linux 上通过 Docker 构建 Windows 包并同步

在仓库**根目录**执行（脚本内会挂载仓库、使用 `electronuserland/builder:wine`，并可能清空 `gibh-desktop-app/node_modules` / `dist-out`，完成后需在宿主机重新 `npm ci`）：

```bash
cd /home/ubuntu/GIBH-AGENT-V2
bash gibh-desktop-app/scripts/dist-win-via-docker.sh
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm ci
```

### 4.6 当前操作系统默认构建（多为单机一种平台）并同步

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist
```

### 4.7 开发调试（不产出发行安装包）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm start
```

## 5. 产物路径说明


| 路径                                                  | 说明                                                                   |
| --------------------------------------------------- | -------------------------------------------------------------------- |
| `gibh-desktop-app/dist-out/`                        | electron-builder 输出目录（`.exe`、`AppImage`、`latest*.yml`、`.blockmap` 等） |
| `services/nginx/html/downloads/`                    | Nginx 静态站点下载目录；下载页 `index.html` 与 `client-release.json` 引用此目录下的文件名   |
| `services/nginx/html/downloads/client-release.json` | 版本号、展示版本字符串、Windows/Linux 安装包**确切文件名**                               |

## 6. 部署到服务器后的运维动作

静态文件更新后，在部署机执行（**compose 服务名请按实际修改**）：

```bash
cd /home/ubuntu/GIBH-AGENT-V2
docker compose restart nginx
```

无 Docker 时：

```bash
sudo nginx -s reload
```

若二进制未放入 Git，需单独同步，示例（把 `USER`、`HOST`、`REMOTE_PATH` 换成实际值）：

```bash
rsync -avz --progress \
  /home/ubuntu/GIBH-AGENT-V2/services/nginx/html/downloads/ \
  USER@HOST:REMOTE_PATH/services/nginx/html/downloads/
```

## 7. Git 与二进制（摘要）

- 纳入版本控制的通常是：`**package.json**`、`**downloads/index.html**`、`**downloads/client-release.json**` 等文本。
- **安装包与 yml / blockmap** 多数规则下 **忽略提交**；发版流水线应在构建完成后把 `**downloads/` 目录内容** 部署到与线上一致的 Nginx 路径。
