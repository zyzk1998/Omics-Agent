# 客户端版本迭代 — 运维全流程说明

本文说明从 **代码修改** 到 **Windows / Linux 安装包构建**，再到 **前端下载页与自动更新元数据同步** 的典型做法，以及可直接执行的命令（假定仓库根目录为 `/home/ubuntu/GIBH-AGENT-V2`）。

---

## 1. 流程总览（运维视角）


| 阶段              | 做什么                                                                                                                                                                                                                             | 产出 / 落点                                 |
| --------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------- |
| 改代码             | 编辑 `main.js`、`package.json` 等                                                                                                                                                                                                   | 本地工作区                                   |
| 升版本号            | 修改 `gibh-desktop-app/package.json` 的 `version`（建议同步 `description`）                                                                                                                                                              | 语义化版本如 `0.12.3`                         |
| 配置发布地址（可选但强烈建议） | ① 仓库根 `.env` 中 `PROD_SERVER_URL`（打进安装包的 Web/API 基址）；② `package.json` → `build.publish[0].url` 中 `**YOUR_SERVER_URL`** 替换为公网可访问的站点根（自动更新 generic 源）；③ 或在运行环境配置 `OMICS_AUTO_UPDATE_BASE_URL` / `OMICS_AGENT_WEB_URL`（见 `main.js`） | 客户端连得上网页与更新服务器                          |
| 准备打包环境          | `npm run prepare-pack-env`：从根目录 `.env` 读取 `PROD_SERVER_URL` 写入 `dist-pack.env`（**不会**把完整 `.env` 打入包，避免密钥泄露）                                                                                                                     | `gibh-desktop-app/dist-pack.env`        |
| 构建安装包           | `electron-builder` 打 Windows（NSIS `.exe`）与/或 Linux（AppImage）                                                                                                                                                                    | `gibh-desktop-app/dist-out/`            |
| 同步到 Nginx 静态站   | `npm run sync:download-page`（已包含在各 `dist*` 脚本末尾）                                                                                                                                                                                | 写入并复制到 `services/nginx/html/downloads/` |
| 部署上线            | 将 `downloads/` 中实际文件与 Git 中已跟踪的元数据一同发布到 Nginx 根；重载 Nginx                                                                                                                                                                        | 用户可访问 `/downloads/` 与自动更新 yml           |


**注意**：`services/nginx/html/downloads/` 下部分二进制（`*.exe`、`*.AppImage`、`latest*.yml`、`.blockmap` 等）通常被 `**.gitignore` 排除**，不进入 Git；**生产环境**需用 **rsync / scp / 发布流水线** 把构建产物同步到服务器，与仓库内 `client-release.json`、`index.html` 一致。

## 2. 发版前检查清单

- `gibh-desktop-app/package.json` 中 `version` 已更新。
- 根目录 `.env` 中 `PROD_SERVER_URL` 指向**目标环境**后端/网页基址（勿把含密钥的整文件复制进客户端资源）。
- `build.publish[0].url` 中占位 `**YOUR_SERVER_URL`** 已替换为真实域名或合规发布地址（参见该文件内嵌 URL 注释说明）。
- 计划同步 `**latest.yml` / `latest-linux.yml`** 及 `***.blockmap`**（脚本会自动从 `dist-out` 复制）；缺失则 **electron-updater** 无法正常工作。
- 按需更新 `services/nginx/html/downloads/index.html` 中的 **Changelog**（纯运维可选步骤）。

## 3. 依赖与环境

- **Node.js + npm**：用于安装依赖与执行脚本。
- **Linux 本机构建 Windows 安装包**：需要 **Wine**，或使用仓库提供的 **Docker** 脚本（推荐在无 Wine 环境时使用）。

首次或清理后安装依赖：

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm ci
```

## 4. 可执行命令（按场景）

以下均在 `**gibh-desktop-app**` 目录下执行（先 `cd`）。

### 4.1 仅生成打包用环境片段（一般不单独执行）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run prepare-pack-env
```

### 4.2 仅同步安装包与元数据到前端 downloads（不打新包）

当 `dist-out/` 里已有与 `package.json` 版本一致的产物时：

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run sync:download-page
```

作用概要：写入 `services/nginx/html/downloads/client-release.json`，复制匹配的 `.exe` / `.AppImage`，复制 `latest.yml`、`latest-linux.yml`、`*.blockmap`，复制 `client-app-icon.png`，并清理同名旧版本安装包文件。

### 4.3 仅构建 Linux（x64 AppImage）并同步

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist:linux
```

### 4.4 仅构建 Windows（x64 NSIS）并同步（本机需 Wine；Linux 无 Wine 会失败）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist:win
```

### 4.5 在无 Wine 的 Linux 上通过 Docker 构建 Windows 包并同步

在仓库**根目录**执行（脚本内会挂载仓库、使用 `electronuserland/builder:wine`，并可能清空 `gibh-desktop-app/node_modules` / `dist-out`，完成后需在宿主机重新 `npm ci`）：

```bash
cd /home/ubuntu/GIBH-AGENT-V2
bash gibh-desktop-app/scripts/dist-win-via-docker.sh
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm ci
```

### 4.6 当前操作系统默认构建（多为单机一种平台）并同步

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm run dist
```

### 4.7 开发调试（不产出发行安装包）

```bash
cd /home/ubuntu/GIBH-AGENT-V2/gibh-desktop-app
npm start
```

## 5. 产物路径说明


| 路径                                                  | 说明                                                                   |
| --------------------------------------------------- | -------------------------------------------------------------------- |
| `gibh-desktop-app/dist-out/`                        | electron-builder 输出目录（`.exe`、`AppImage`、`latest*.yml`、`.blockmap` 等） |
| `services/nginx/html/downloads/`                    | Nginx 静态站点下载目录；下载页 `index.html` 与 `client-release.json` 引用此目录下的文件名   |
| `services/nginx/html/downloads/client-release.json` | 版本号、展示版本字符串、Windows/Linux 安装包**确切文件名**                               |


---

## 6. 部署到服务器后的运维动作

静态文件更新后，在部署机执行（**compose 服务名请按实际修改**）：

```bash
cd /home/ubuntu/GIBH-AGENT-V2
docker compose restart nginx
```

无 Docker 时：

```bash
sudo nginx -s reload
```

若二进制未放入 Git，需单独同步，示例（把 `USER`、`HOST`、`REMOTE_PATH` 换成实际值）：

```bash
rsync -avz --progress \
  /home/ubuntu/GIBH-AGENT-V2/services/nginx/html/downloads/ \
  USER@HOST:REMOTE_PATH/services/nginx/html/downloads/
```

---

## 7. Git 与二进制（摘要）

- 纳入版本控制的通常是：`**package.json**`、`**downloads/index.html**`、`**downloads/client-release.json**` 等文本。
- **安装包与 yml / blockmap** 多数规则下 **忽略提交**；发版流水线应在构建完成后把 `**downloads/` 目录内容** 部署到与线上一致的 Nginx 路径。
