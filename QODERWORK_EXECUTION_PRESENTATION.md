# QoderWork 类智能体平台：执行记录呈现与实现说明

本文档根据 **QoderWork**（或同类「自主执行环境配置」的智能体产品）在 **GIBH Agent 桌面客户端部署** 等任务中的界面表现，归纳其 **思考过程、执行过程、执行记录、任务监控** 的 **呈现方式** 与可落地的 **实现方式**，供本仓库对接自动化工具、自研控制台或撰写 PRD 时参考。

---

## 1. 总体架构：四层信息模型

| 层级 | 用户可见名称（示例） | 用户价值 | 典型载体 |
|------|----------------------|----------|----------|
| **思考层** | 深度思考 / Deep Thinking | 解释「为何这样做、遇到了什么约束」 | 主时间线内可折叠块 |
| **动作层** | 步骤列表、「查看 N 个步骤」 | 将目标拆成可核对的大步骤与子步骤 | 时间线 + 展开面板 |
| **日志层** | 正在执行命令…、终端输出 | 可审计、可排障的原始命令与 stdout/stderr | 底部浮条 + 滚动日志区 |
| **监控层** | 任务监控、待办、产物 | 全局进度与产出物一览 | 右侧边栏固定面板 |

实现上可将一次「任务（Task）」建模为：**任务元数据 + 阶段（Phase）列表 + 每条阶段下的事件流（Event）**；事件类型区分 `thought`、`step_start` / `step_end`、`command`、`artifact`、`todo_update` 等，前端按类型路由到上述四层 UI。

---

## 2. 思考层（深度思考）

### 2.1 呈现方式

- 出现在 **主对话/活动流** 中，与「助手回复」同级时间序列。
- 使用 **可折叠区块**（标题如「深度思考」），默认可收起，避免挤占主结论。
- 正文为 **自然语言推理**：依赖是否齐全（如 Node/npm）、环境限制（如访问 GitHub 失败）、对用户输入的纠错（如 `198.168.x.x` → `192.168.x.x`）。
- 作用：在 **执行动作之前或并行** 展示「意图 → 约束 → 策略」，降低黑盒感。

### 2.2 实现方式（建议）

1. **流式输出**：模型侧以 SSE/WebSocket 推送 `reasoning` 或 `thinking` 类型 chunk；前端累积为 Markdown/纯文本，支持打字机效果。
2. **与最终答案分离**：协议上区分 `role: assistant` 下的 `content` 与 `reasoning_content`（或独立 channel），便于折叠与权限控制（部分产品可对思考内容单独脱敏）。
3. **持久化**：同一 `task_id` 下按序号写入事件表或 append-only 日志，支持任务结束后回看。
4. **可选策略**：长思考自动摘要一行展示在折叠标题上（如「检测到缺少 Node，将先安装 LTS」）。

## 3. 动作层（执行过程与步骤）

### 3.1 呈现方式

- **时间顺序**：每个大块动作一条卡片或一条时间线节点（如「创建项目文件」「检查 winget」「安装 Node.js」）。
- **子步骤聚合**：主节点附带 **「查看 N 个步骤」** 链接，点击后展开 **子任务列表**（具体命令、文件操作、检查结果）。
- **状态**：进行中显示「处理中…」与 **加载动画**；完成则展示 **结果摘要**（如「Node.js 安装成功」）。
- **可点击性**：强调「哪一步在跑、哪一步已完成」，便于用户在中断后从某步继续或人工介入。

### 3.2 实现方式（建议）

1. **数据模型**  
   - `Step`：`id`, `parent_id`, `title`, `status`（pending / running / success / failed / skipped）, `order`, `started_at`, `ended_at`。  
   - 子步骤用 `parent_id` 挂树，或扁平列表 + `group_id`。
2. **状态机**：执行器每完成一个原子操作向服务端上报 `step_update`；前端用乐观 UI 或轮询/WebSocket 同步。
3. **「查看 N 个步骤」**：`N = children.length` 或 `sub_events` 计数；展开区懒加载子步骤详情，避免首屏过大。
4. **失败处理**：失败步骤高亮，附带 `error_code`、最后一条命令、建议修复（可由模型生成）。

## 4. 日志层（执行记录 / 命令透明）

### 4.1 呈现方式

- 在主流 **底部** 或 **固定底栏** 展示 **当前/最近一条 shell 命令** 全文（截图示例：`powershell.exe -Command "Set-Location '...'; npm install"`）。
- **浮层状态条**：如绿色条 **「正在执行命令…」**，并显示 **已执行时长**（如 `155.4s`），强化长耗时任务的可见性。
- 可扩展：完整 **终端滚动区**（ANSI 着色、复制、搜索），与「动作层」某一步双向锚定（点击步骤跳转对应日志区间）。

### 4.2 实现方式（建议）

1. **命令执行沙箱**：Agent 在受控环境（本地 daemon、远程 VM、容器）中执行；stdout/stderr **实时流式** 推送到前端（WebSocket 分片或 chunked HTTP）。
2. **结构化元数据**：每条日志附带 `run_id`, `cwd`, `argv`, `exit_code`, `duration_ms`，便于与 `Step` 关联。
3. **安全与合规**：敏感参数脱敏；超长输出截断 + 「下载完整日志」链接。
4. **跨平台**：Windows 展示 `powershell.exe` / `cmd`；Unix 展示 `bash -lc '...'`，由执行层统一封装为「可展示的 command 字符串」。

## 5. 监控层（任务监控：待办 + 产物）

### 5.1 呈现方式（待办）

- **右侧边栏** 固定 **「任务监控」** 面板，与主时间线并列，不随滚动丢失上下文。
- **待办清单**：  
  - **已完成**：绿色勾。  
  - **进行中**：蓝色圆点/高亮。  
  - **未开始**：空心圆。  
- 示例任务链：检查 Node → 创建项目文件 → 执行 `npm install` → 执行 `npm start` → 生成 Windows 安装包。
- 与动作层区别：**待办偏「用户可理解的里程碑」**，动作层偏「实际发生的操作序列」；二者可映射（一个待办对应多个步骤）。

### 5.2 呈现方式（产物 / Artifacts）

- 同侧边栏下方 **「产物」** 区域：列出 **默认工作目录** 下由 Agent 创建或修改的文件（如 `package.json`, `main.js`）。
- 可扩展：文件大小、修改时间、预览链接、下载 zip。

### 5.2 实现方式（建议）

1. **待办同步**：任务创建时由规划器（Planner）生成初始 `Todo[]`；每完成一个里程碑，执行器回调 `PATCH /tasks/:id/todos/:tid` 更新状态；WebSocket 广播 `todo_changed`。
2. **产物索引**：工作区 watcher 或任务结束时的 `git status` / 文件树 diff，生成 `Artifact[]`（path, kind, hash）；大文件只记元数据，内容按需拉取。
3. **权限**：产物路径限制在任务工作根目录内，防止遍历敏感盘符。

## 6. 与主界面其它区域的协同

| 区域 | 作用 |
|------|------|
| **左侧边栏** | 新任务、技能（Skills）、历史任务列表，用于切换上下文。 |
| **底部输入栏** | 显示当前工作目录（如 `D:\omics agent`）、队列输入（「添加到队列」），支持任务中途追加指令。 |
| **用户/套餐信息** | 与执行无关，属账户与计费展示。 |

实现时建议：**全局 `workspaceRoot` + 当前 `taskId`** 注入到底部栏与执行器，保证命令 `cd` 与文件产物路径一致。

---

## 7. 与本仓库（GIBH Agent）的衔接建议

- **桌面客户端路径**：自动化常在 `gibh-desktop-app/` 下执行 `npm ci`、`npm run dist`；文档化时可将「待办模板」固定为该目录下的标准步骤，并在监控层映射。
- **服务地址**：`main.js` 中 `loadURL` 若依赖固定 IP/端口，思考层应提示用户核对网络；日志层应打印实际请求的 URL（便于排查 `ERR_CONNECTION_REFUSED`）。
- **Docker 与 8018**：若任务包含「起全栈」，动作层应包含 `docker compose up` 与 `curl -I http://127.0.0.1:8018` 等可验证子步骤，并在日志层展示完整命令。

## 8. 附录：最小事件协议示例（便于前后端对齐）

以下为示意 JSON，非 QoderWork 官方接口。

```json
{
  "task_id": "uuid",
  "events": [
    { "type": "thought", "ts": "...", "body": "检测到未安装 Node，将使用 winget 安装 LTS。" },
    { "type": "step_group", "id": "sg1", "title": "安装 Node.js", "status": "running" },
    { "type": "command", "step_id": "sg1", "ts": "...", "shell": "powershell", "line": "winget install OpenJS.NodeJS.LTS" },
    { "type": "log_chunk", "run_id": "r1", "stream": "stdout", "text": "..." },
    { "type": "todo", "items": [
      { "id": "t1", "title": "检查 Node.js", "status": "done" },
      { "id": "t2", "title": "npm install", "status": "running" }
    ]},
    { "type": "artifact", "path": "gibh-desktop-app/package.json", "action": "modified" }
  ]
}
```
