📋 全流程数据流向详解

步骤 1: 用户交互与文件上传

  1. 用户操作：用户在 Web UI 中选择文件或输入查询文本
  2. 文件上传：前端调用 POST /api/upload，上传文件到服务器
  3. 文件处理：
    • 服务器自动识别 10x Genomics 数据（matrix.mtx, barcodes.tsv, features.tsv）
    • 文件保存到 {user_id}/{session_id}/ 目录
    • 生成文件元数据（文件类型、样本数、特征数等）
  4. 返回路径：服务器返回 file_paths 数组和 session_id

数据格式：文件上传响应

{ "status": "success", "file_paths": ["guest/20250128_120000/example.csv"], "file_info": [{"name": "example.csv", "size": 1024000, "path": "..."}], "session_id": "20250128_120000", "user_id": "guest" }

步骤 2: 聊天请求与意图分类

  1. 发送请求：前端调用 POST /api/chat，包含：
    • message：用户查询文本
    • uploaded_files：已上传的文件列表（name, path）
    • stream: true：启用 SSE 流式响应
    • user_id 和 session_id：用户和会话标识
  2. API 路由：FastAPI 服务器接收请求，创建 AgentOrchestrator 实例
  3. 意图分类：AgentOrchestrator.stream_process() 首先进行全局意图分类：
    • Chat 模式：如果查询是普通对话（无文件、无分析关键词），直接调用 LLM 返回回答
    • Task 模式：如果查询包含分析任务，进入工作流规划和执行流程

数据格式：聊天请求

{ "message": "分析这个文件", "uploaded_files": [ {"name": "example.csv", "path": "guest/20250128_120000/example.csv"} ], "stream": true, "user_id": "guest", "session_id": "20250128_120000" }

步骤 3: 文件检查与数据诊断（Task 模式）

  1. 文件检查：调用 FileInspector 工具检查上传的文件
    • 识别文件类型（CSV, H5AD, MTX 等）
    • 提取基本统计信息（样本数、特征数、缺失值比例等）
    • 生成数据预览
  2. 数据诊断：可选的数据质量评估
    • 调用 LLM 生成数据诊断报告
    • 识别潜在问题（缺失值、异常值、数据分布等）
    • 通过 SSE 事件 diagnosis 发送到前端

步骤 4: 动态工作流规划

  1. 工具检索：WorkflowPlanner 调用 ToolRetriever
    • 使用 ChromaDB 向量数据库进行语义检索
    • 根据用户查询和文件类型，检索相关的分析工具
    • 返回工具的 JSON Schema（参数定义、描述等）
  2. LLM 规划：将工具 Schema 注入到 LLM 提示词中
    • LLM 根据用户查询和可用工具，生成工作流计划
    • 输出 JSON 格式的工作流配置：{workflow_name, steps: [{step_id, tool_id, name, params}]}
    • 每个步骤包含工具ID和参数配置
  3. 参数推荐：可选步骤，LLM 根据数据特征推荐最佳参数
    • 分析数据统计信息
    • 生成参数推荐（如缺失值阈值、标准化方法等）
    • 填充到工作流步骤的 params 中
  4. 返回工作流：通过 SSE 事件 workflow 发送工作流配置到前端

数据格式：工作流配置

{ "type": "workflow", "workflow_config": { "workflow_name": "Metabolomics Analysis Pipeline", "steps": [ { "step_id": "inspect_data", "tool_id": "inspect_data", "name": "数据检查", "params": {"file_path": "example.csv"} }, { "step_id": "preprocess_data", "tool_id": "preprocess_data", "name": "数据预处理", "params": { "file_path": "example.csv", "missing_threshold": "0.5", "normalization": "log2" } } ] }, "template_mode": false }

步骤 5: 工作流执行与工具调用

  1. 用户确认：前端显示工作流配置卡片，用户点击"执行"按钮
  2. 执行请求：前端发送 workflow_data 到 /api/chat 或 /api/execute
  3. 路由到领域智能体：
    • RouterAgent 根据 workflow_name 或文件类型，路由到对应的领域智能体（RNA Agent 或 Metabolomics Agent）
    • 领域智能体负责执行特定组学类型的分析
  4. 创建执行器：领域智能体创建 WorkflowExecutor 实例
  5. 逐步执行：WorkflowExecutor 按顺序执行每个步骤
    • 从 ToolRegistry 查找工具（通过 tool_id）
    • 使用 Pydantic 验证步骤参数
    • 调用工具函数，传入参数
    • 收集工具返回的结果
    • 处理步骤间的数据流（如将前一步的输出作为下一步的输入）
  6. 实时更新：每个步骤执行完成后，通过 SSE 事件 step_result 发送结果到前端

数据格式：步骤执行结果

{ "type": "step_result", "report_data": { "step_id": "inspect_data", "tool_id": "inspect_data", "name": "数据检查", "summary": "检查完成: 77 个样本, 50 个代谢物", "status": "success", "plot": "/results/run_20250128_120000/inspect_plot.png", "step_result": { "step_name": "数据检查", "status": "success", "logs": "检查完成: 77 个样本, 50 个代谢物", "data": { "summary": { "n_samples": 77, "n_metabolites": 50, "missing_percentage": 2.5 }, "preview": [...], "images": ["/results/run_20250128_120000/inspect_plot.png"] } } } }

步骤 6: 结果聚合与格式化

  1. 结果收集：WorkflowExecutor 收集所有步骤的执行结果
  2. 数据格式化：
    • 统一结果格式（steps_details, steps_results）
    • 处理图片路径（转换为可访问的 URL，如 /results/run_xxx/plot.png）
    • 提取关键指标（样本数、特征数、聚类数等）
  3. 生成最终报告：构建完整的分析报告数据结构

步骤 7: AI 专家分析报告生成

  1. 提取关键指标：BaseAgent._generate_analysis_summary() 从步骤结果中提取关键指标
    • RNA 分析：细胞数、基因数、线粒体比例、标记基因、聚类数、细胞类型等
    • 代谢组学：样本数、代谢物数、差异代谢物、VIP 值、通路富集等
  2. 构建提示词：
    • 精简步骤信息（只保留核心字段，过滤冗余数据）
    • 限制列表长度（如只保留 top 3-5 个标记基因/通路）
    • 检查提示词长度，防止超出 LLM 上下文窗口
  3. LLM 生成报告：调用 LLM 生成 Markdown 格式的专家分析报告
    • 包含分析结果摘要
    • 关键发现和生物学解释
    • 数据质量评估
    • 建议和下一步分析方向
  4. 返回报告：通过 SSE 事件 diagnosis 发送 AI 专家分析报告到前端

数据格式：AI 专家分析报告

{ "type": "diagnosis", "report_data": { "diagnosis": "## AI 专家分析报告\n\n### 分析结果摘要\n\n本次分析完成了 5 个步骤...\n\n### 关键发现\n\n- 成功步骤: 5/5\n- 细胞数量: 2,700\n- 基因数量: 3,274\n..." } }

步骤 8: SSE 流式输出与前端渲染

  1. SSE 事件流：AgentOrchestrator 通过 yield 生成 SSE 事件
    • status：状态更新（"正在执行步骤...", "正在生成报告..."）
    • workflow：工作流配置
    • step_result：步骤执行结果
    • diagnosis：AI 专家分析报告
    • result：最终结果
    • done：完成信号
    • error：错误信息
  2. 前端接收：前端使用 ReadableStream 读取 SSE 流
    • 解析 SSE 格式（event: {type}\ndata: {json}\n\n）
    • 根据事件类型调用相应的处理函数
  3. 实时渲染：
    • status → 更新状态栏
    • workflow → 渲染工作流配置卡片
    • step_result → 渲染步骤执行结果（图表、表格）
    • diagnosis → 渲染 AI 专家分析报告（Markdown）
    • result → 渲染最终分析报告

SSE 事件格式示例

event: status data: {"content": "正在执行步骤: 数据检查...", "state": "running"} event: step_result data: {"report_data": {...}} event: diagnosis data: {"report_data": {"diagnosis": "## AI 专家分析报告\n\n..."}} event: done data: {"status": "success"}

🔑 关键技术点

1. 工具自动发现与注册

系统通过 Python 装饰器和反射机制，自动发现和注册工具到 ToolRegistry。新增工具只需创建文件并添加 @registry.register 装饰器，无需修改核心代码。

2. 语义工具检索

使用 ChromaDB 向量数据库和 Ollama Embeddings，实现基于语义相似度的工具推荐。用户可以用自然语言描述需求，系统自动推荐相关工具。

3. LLM 驱动的工作流规划

将工具 Schema 注入到 LLM 提示词中，LLM 根据用户查询和可用工具，动态生成可执行的工作流计划。无需硬编码模板，支持灵活的分析流程。

4. 通用执行引擎

基于策略模式和依赖注入，实现工具无关的执行框架。支持动态工具查找、参数验证（Pydantic）和结果聚合，无需为每个工具编写特定的执行逻辑。

5. SSE 流式输出

使用 Server-Sent Events (SSE) 实现实时流式输出，前端可以实时接收和渲染分析进度、步骤结果和 AI 报告，提升用户体验。

6. 多用户隔离

支持多用户并发使用，每个用户有独立的文件目录（{user_id}/{session_id}/），工作流和任务历史按用户隔离。

📊 数据流总结

完整数据流向

  1. 前端 → API 服务器：HTTP POST 请求（文件上传、聊天请求）
  2. API 服务器 → 编排器：创建 AgentOrchestrator，调用 stream_process()
  3. 编排器 → 智能体：意图分类、路由到领域智能体、调用规划器
  4. 规划器 → 工具检索器/LLM：语义检索工具、LLM 生成工作流
  5. 领域智能体 → 执行器：创建 WorkflowExecutor，执行工作流
  6. 执行器 → 工具：动态查找工具、验证参数、调用工具函数
  7. 工具 → 执行器：返回执行结果（数据、图表路径、统计信息）
  8. 执行器 → 领域智能体：聚合步骤结果、生成 AI 报告
  9. 领域智能体 → 编排器：返回完整分析报告
  10. 编排器 → API 服务器：通过 SSE 流式输出事件
  11. API 服务器 → 前端：SSE 事件流（status, workflow, step_result, diagnosis, result, done）
  12. 前端渲染：实时解析 SSE 事件，渲染工作流卡片、步骤结果、AI 报告
