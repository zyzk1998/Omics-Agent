# 参数推荐功能 — 开发与维护指南

本文档描述 GIBH-AGENT-V2 中「数据诊断 → 参数推荐 Markdown 表 → 后端解析 → 一键回填工作流卡片」的**前后端字段契约**、**白名单与可调参数提取**、以及**新技能 / 新工作流接入**时的检查清单。架构原则仍以 `docs/ARCHITECTURE_CORE_PRINCIPLES.md` 与 `.cursor/rules/gibh-architecture-constitution.mdc` 为准。

---

## 1. 前后端字段映射原理

### 1.1 表单控件命名（唯一真源）

- 工作流卡片由 `renderWorkflowForm`（`services/nginx/html/index.html`）根据 `workflow_data.steps[].params` 渲染。
- 每个可编辑字段的 DOM 名称为：

  `name="param_{步骤索引}_{参数Key}"`

  其中 **`参数Key`** 与 Python 工作流模板 / 工具参数字典的键一致（多为 `snake_case`），与执行器反射注入所用名称一致。

- 文本类参数输入框额外带有 `data-param-name="{参数Key}"`，供「一键应用」在 Key 被轻微污染时仍能定位。

### 1.2 诊断报告中的 Markdown 表

- `BaseAgent._perform_data_diagnosis` 调用 LLM 生成报告，其中「### 💡 参数推荐」小节为 Markdown 表格。
- **第一列必须是上述 `参数Key`**，与 `param_{i}_{key}` 中的 `key` **逐字相等**（大小写、下划线均不可改）。
- 第二～四列依次为：默认值、推荐值、推荐理由（与 `_extract_parameter_recommendations` 解析逻辑一致）。

### 1.3 后端解析 → 前端消费

1. `BaseAgent._extract_parameter_recommendations` 用正则截取 `### 💡 参数推荐` 后的表格，解析第一列为 Key，第三列为推荐值。
2. 若传入 `allowed_param_names`（由白名单得到的 `frozenset`），**不在集合内的行丢弃**，避免幻觉键进入 JSON。
3. 结果写入 `self.context["parameter_recommendation"]`，编排器随 `workflow` / `diagnosis` SSE 字段 `recommendation` 下发。
4. 前端将 `recommendation` 存入 `window.workflowRecommendations[formId]`。
5. `applyRecommendedParams(formId, stepsLength)` **不重新解析 Markdown**，只遍历 `rec.params`，按下列顺序匹配 DOM：
   - `name="{cleanKey}"`
   - 将空格类标题归一为 `snake_case` 后再按 `name` 查找
   - `.param-input[data-param-name="..."]`（优先兼容 LLM 输出与 UI 标签混淆）
   - `name="param_{i}_{cleanKey}"`（`i` 从 0 到 `stepsLength`）
   - 支持 `"{stepIndex}:{key}"` 形式的复合键（便于后续扩展）

## 2. 白名单提取逻辑

### 2.1 `diagnosis_param_whitelist.py`

| 函数 | 作用 |
|------|------|
| `build_diagnosis_whitelist_prompt(workflow, target_step_ids, file_metadata)` | 调用 `workflow.generate_template(...)`，遍历 `steps[].params`，生成 **步骤级 Valid Keys** 与 **【致命约束】** 提示块；并附 `humanize_param_key_as_ui_label` 供模型区分「Key」与「人类可读标签」 |
| `collect_param_whitelist_for_diagnosis(...)` | 模板失败时回退 `steps_dag` + `get_step_metadata` 的 `default_params` 键 |

### 2.2 `BaseAgent._extract_available_parameters`

- 封装对上述函数的调用；若模板无键则回退收集扁平 Key，并拼接 `_format_diagnosis_param_whitelist_prompt`。
- **`_perform_data_diagnosis` 在已通过 `workflow_for_whitelist` 注入长文约束时，不得再用「仅扁平 Key」块覆盖**，否则会丢失步骤级说明（实现上已避免二次覆盖）。

### 2.3 `BaseAgent._extract_editable_params`（结构化）

- 入参：`workflow`、`target_step_ids`、`file_metadata`（与诊断白名单一致）。
- 返回：`List[{"step_index","step_id","step_name","param_key","label"}]`，用于联调、日志、或后续 UI/Agent 扩展。
- 遍历规则与 `build_diagnosis_whitelist_prompt` 对齐；无 `steps` 时回退 DAG 元数据。

### 2.4 调用方约定

- **编排器 Path A**、各域 Agent 在调用 `_perform_data_diagnosis` 时应传入 **`workflow_for_whitelist`** 与尽量准确的 **`target_step_ids_for_whitelist`**，与规划步骤一致，避免白名单过大或缺项。

## 3. 语义路由与药物相似性技能（`drug_similarity`）

- `gibh_agent/core/semantic_router.py` 中，当用户意图为 **药物相似性 / Lipinski / 结构相似检索** 等**可执行技能**（非纯百科）时，路由为 **`skill_fast_lane`**，且 `rationale_short` 须包含 **`skill_id=drug_similarity`**，供审计与编排器识别。
- 编排器在无 `[Skill_Route: ...]` 暗号时，可根据 `skill_id=drug_similarity` 与查询关键词，在 **`lipinski_druglikeness`** 与 **`drug_similarity_search`** 之间自动补全暗号并调用 `SkillAgent`。
- 前端技能广场通过 **「药物相似性评估工具」** 卡片与 **二选一 Modal** 向输入框注入带 `[Skill_Route: ...]` 的引导词，与上述后端契约一致。

## 4. 新组学 / 新工作流接入规范

1. 工作流类继承 `BaseWorkflow`，`get_step_metadata` 中 **`default_params` 的键集** 必须与前端卡片、工具签名一致。
2. `generate_template` 返回的 `workflow_data.steps[].params` 与运行时一致；**仅用户可调**的键放入模板 `params`。
3. 新模态 Agent 调用 `_perform_data_diagnosis(..., workflow_for_whitelist=..., target_step_ids_for_whitelist=...)`。
4. 若改动 SSE、`state_snapshot`、历史恢复或推荐结构，遵守宪法「时光机四联」与本文 1～2 节字段契约。

---

## 5. 故障排查

| 现象 | 排查 |
|------|------|
| 一键应用 0 条 | 浏览器控制台 `[applyRecommendedParams]`；核对表格第一列 Key 与 `param_{i}_{key}`；核对 `data-param-name` |
| 推荐表出现未在表单中的 Key | 白名单是否注入成功；是否误覆盖 `whitelist_prompt_extra` |
| 诊断内容陈旧 | 删除侧车诊断缓存或更换输入路径重跑 |

## 6. 维护记录

- **动态步骤级白名单**：`build_diagnosis_whitelist_prompt`、`BaseAgent._extract_available_parameters`、编排器 / 域 Agent 传入 `workflow_for_whitelist`。
- **结构化可调参**：`BaseAgent._extract_editable_params`。
- **语义快车道药物技能**：`SemanticRouter` 文案、`orchestrator` 补全 `[Skill_Route]`、`SkillAgent` 双模式 persona、前端 Modal + `applyRecommendedParams` 增强。
