阶段 1 · 探索期

agent-mini：先证明「闭环存在」

零框架 HTTP + 工具 JSON —— 快，但不可规模化。

技术指纹
curl / bash Ollama REST tools.json ReAct 四步
主要问题

一条脚本顶全家：没会话、没文件资产、也没并发隔离。

线上出问题基本靠猜，日志和指标都薄。

当时证明了什么

模型能按 JSON 点名工具，HTTP 真执行，再把结果塞回对话里——闭环跑得通。

结论：路子对，但不能停在脚本层。

上图是公开原型 agent-mini 一类结构：没有重型框架，靠 tools.json 把「可调函数」交给模型，适合验证，不适合直接当生产底座。

阶段 2 · 阵痛期

vLLM + 固定流水线：第一次「像产品」

代表形态 · 技术栈
Docker Compose FastAPI Celery Redis 双 vLLM 逻辑 / 多模态分流
问题 A · 意图被带偏

靠关键词触发流水线，「查个基因」也能误开整套 scRNA。

用户多等一轮，GPU 白烧一轮。

人和机器都吃亏

问题 B · 推理和重活抢一条路

聊天推理和重计算走同一套服务链。

高峰时 API 先扛不住，排障还难拆。

扩容也治标不治本

结论：光靠关键词不够，要把「该不该跑重活」先判清楚——也就是后面的结构化路由。

阶段 3 · 规划

Task 规划：总控 × 首席科学家 × 图书管理员

与「模块交互总览」同一套紫青视觉语言：先判意图、拆科学问题、再对齐工具库与 SOP，最后由 LLM 产出可执行工作流草案。

阶段 3 · 执行

Task 执行：领域专家 × 执行引擎 × SSE

工作流卡片确认后进入执行链：Executor 调原子工具与沙箱，错误经红路回灌 LLM 做修正；结果经 SSE 推送到前端实时渲染。

核心壁垒 ①

结构化路由：静态红线 + 动态护栏

gibh_agent/core/semantic_router.py —— 先判「该不该动手」，再谈「动哪只手」。

静态 A–E

没上传就别启动重管线；没开算力通道就别走超算；明显百科问答就别硬套 Pipeline。

动态 F · digest

把当前环境里真实存在的 SOP 名、工具 id 摘要给模型，减少「编一个不存在的工具名」。

硬兜底

解析 JSON → Pydantic 校验 → 置信度低就重试 → 仍不行就追问用户，避免悄悄走错分支。

Chapter 04 · 价值与规划

商业叙事：高可用 × 数据飞轮

说人话版：系统稳定跑起来以后，把真实对话里踩过的坑留下来，反过来改路由和工具说明，下次少浪费算力、少返工。

Q1 · 把接口和评测钉住
eval 脚本进常规回归；路由与工具 schema 带版本号；功能开关支持灰度。
Q2 · 让数据转起来
脱敏日志里挖难例 → 改 prompt / digest → 再看线上指标是否变好。
Q3+ · 多团队与额度
配额、队列优先级、超算预算面板；桌面端和私有化同一套交付形态。
Q4 · 科研数据与语料衔接
把「分析产出」和「数据汇交、语料沉淀」串成可自动化的链路，服务国家科研基础设施方向——落地节奏与合规边界需单独评审。
