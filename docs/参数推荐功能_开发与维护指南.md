# 参数推荐功能 — 开发与维护指南

本文说明 GIBH-AGENT-V2 中「数据诊断报告 → 参数推荐 → 一键应用」的全链路约定，以及 **动态参数白名单** 的实现与扩展规范，适用于 RNA / 代谢组学 / 影像组学 / 空间转录组等所有通过 `BaseWorkflow` + 前端工作流卡片承载的模态。

---

## 1. 一键应用：前后端交互原理

### 1.1 后端：Markdown 报告 → 结构化推荐

1. **统计事实**：`DataDiagnostician.analyze()` 产出结构化 stats（无 LLM 幻觉约束主体）。
2. **LLM 报告**：`BaseAgent._perform_data_diagnosis()` 调用 LLM，生成含「### 💡 参数推荐」章节的 **Markdown**。
3. **表格解析**：`BaseAgent._extract_parameter_recommendations()` 用正则截取 `### 💡 参数推荐` 后的 **Markdown 表格**，按列解析：
   - 第 1 列：参数 **Key**（必须与前端表单字段一致）
   - 第 2 列：默认值（展示用）
   - 第 3 列：推荐值
   - 第 4 列：理由  
4. **白名单过滤**：若本次诊断注入了 `allowed_param_names`（`frozenset`），**不在白名单内的行会被丢弃**，避免捏造字段进入 `recommendation.params`。
5. **下发前端**：编排器将 `recommendation` 与 `diagnosis_report` 一并写入工作流 SSE / `workflow_data`（与 `planner` 产出的 `recommended_params` 路径并存时以前端消费结构为准）。

### 1.2 前端：状态回填（非「正则解析 Markdown」）

**重要**：浏览器 **不会** 从 Markdown 里再正则拆表；它只消费后端已解析好的 JSON：

- `renderWorkflowForm` 将 `recommendation` 存到 `window.workflowRecommendations[formId]`。
- 输入框命名规则：`name="param_{步骤索引}_{参数Key}"`，其中 **`参数Key`** 来自各步 `step.params` 的对象键（与后端工具/执行器一致，多为 `snake_case`）。
- 「✨ 一键应用推荐参数」调用 `applyRecommendedParams(formId, stepsLength)`：对 `rec.params` 的 **每个 Key** 依次尝试：
  - `[name="{Key}"]`
  - `[name="param_{i}_{Key}"]`（`i` 从 0 遍历到 `stepsLength`）

因此：**表格第一列必须是上述 Key 字符串**，与 UI 上的英文标签（如 `Min Genes`）**不是**同一字符串；UI 标签由前端对 Key 做展示化（下划线转空格 + 首字母大写）。

### 1.3 防守层

- **后端**：剥离第一列的反引号/加粗；白名单过滤。  
- **前端**：`applyRecommendedParams` 对 Key 做首尾反引号/星号清理，降低 LLM 轻微格式污染导致的匹配失败。

## 2. 动态白名单（全局强约束）

### 2.1 目标

让 LLM 在写参数推荐表时，**只能**使用「当前工作流模板里真实出现的、会渲染成表单控件的参数字段」，避免出现例如 `log_transform` 等 **卡片上不存在** 的键，从而导致一键应用无法回填。

### 2.2 实现要点

| 组件 | 路径 | 职责 |
|------|------|------|
| 步骤级白名单与 Prompt 正文 | `gibh_agent/core/diagnosis_param_whitelist.py` | `build_diagnosis_whitelist_prompt(workflow, target_step_ids, file_metadata)`：基于 `workflow.generate_template(...)` 遍历 `steps[].params`，按步骤输出 `Step: … -> Valid Keys: […]`，并附 **UI 标签参考**（`humanize_param_key_as_ui_label`，与前端展示规则对齐）；返回 **扁平 Key 列表** + **【致命约束：参数推荐规则】** 文本块 |
| 父类统一入口 | `gibh_agent/agents/base_agent.py` | `_extract_available_parameters(...)` 调用上述函数；若模板无键则回退 `collect_param_whitelist_for_diagnosis` + 扁平约束 |
| 诊断 LLM 拼装 | 同上 `_perform_data_diagnosis` | 当传入 `workflow_for_whitelist` 时，用 `_extract_available_parameters` 得到 **Keys** 与 **注入块**，写入 **system + user** prompt；`_extract_parameter_recommendations(..., allowed_param_names=...)` 与 Keys 对齐 |
| 编排主路径 | `gibh_agent/core/orchestrator.py` | Path A 有文件诊断时传入 `workflow_for_whitelist=workflow`、`target_step_ids_for_whitelist=target_steps`，**不再**手工拼扁平列表 |
| 领域 Agent | `rna_agent.py` / `metabolomics_agent.py` 等 | 通过 `WorkflowRegistry().get_workflow(...)` 传入同一参数，规划结果存在时尽量传入 **当前 plan 的 step_id 序列** 以裁剪模板 |

### 2.3 Prompt 中的「致命约束」摘要（语义）

- 表格第一列 **必须** 为上文列出的 **Key** 之一，**一字不差**（大小写、`snake_case` 均不可改）。  
- **禁止**用 UI 中文名、Title Case 替代 Key。  
- **禁止**为第一列加 `**` 或反引号。  
- 若无法遵守：**宁可不写参数表**，仅在正文定性描述。

## 3. 新组学 / 新工作流接入规范（空间转录组等）

1. **工作流类**继承 `BaseWorkflow`，实现 `get_steps_dag`、`get_step_metadata`，且 **`default_params` 的键集 = 前端将要生成的 `step.params` 键集**（与执行器工具签名一致，遵守架构宪法中的路径参数命名表）。  
2. **`generate_template`**：确保返回的 `workflow_data.steps[].params` 与真实执行卡片一致；新增可调参数必须出现在 **至少一步** 的 `params` 字典中，否则 LLM 白名单里不会出现，也无法一键应用。  
3. **若某参数仅内部使用、不对用户开放**：不要放进 `default_params` / 模板 `params`，否则会被白名单放行但产品上不期望展示——应通过工具默认值或执行器注入处理。  
4. **诊断入口**：新模态 Agent 在调用 `_perform_data_diagnosis` 时务必传入 **`workflow_for_whitelist`**（及可选 **`target_step_ids_for_whitelist`**），与 Orchestrator 主路径行为一致。  
5. **联调自检**：改 SSE / 入库快照 / 历史恢复时仍须遵守 `.cursor/rules/gibh-architecture-constitution.mdc` §5（时光机四联）；改参数推荐时建议额外核对本节表格中的四列解析与前端 `param_{i}_{key}`。

## 4. 故障排查清单

| 现象 | 可能原因 | 处理 |
|------|-----------|------|
| 一键应用 0 个字段 | `recommendation.params` 的 Key 不在表单 `name` 中 | 查 LLM 表格第一列是否偏离白名单；查该步 `params` 是否含该键 |
| 报告仍出现幻觉参数名 | 缓存 `.diagnosis.json` 旧结果 | 删除侧车缓存或换输入重跑 |
| 白名单为空 | `generate_template` 异常或 steps 无 params | 查日志 `build_diagnosis_whitelist_prompt`；检查 DAG 元数据 |
