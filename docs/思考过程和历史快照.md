## 前端多任务状态机与视图穿透攻坚战

> **背景**：技能广场「使用中」会话在后台跑 SSE 时，用户切去技能广场 / 历史 / 新建会话，曾出现 **高亮死锁**（侧栏卡片底色永不消退）、**误杀流**（`abort()` 或 `innerHTML=''` 清掉活 DOM）、**连续点击拦截**等问题。  
> **终局方案**：**DOM 物理隔离 + 双轨路由（Dual-Layer Routing）**，见 `services/nginx/html/index.html` 与 `docs/持久化层构建规范.md` § 10。

### 1. 演进路径：从「逻辑切换」到「物理隔离」

| 阶段 | 做法 | 典型故障 |
|------|------|----------|
| **V0 单容器** | 仅 `#chat-main-host` / `#chatContent`；切视图时 `loadSessionAndRestore` 或 `clearChatAndReset` 清 DOM | 后台 SSE 仍在写已被销毁的节点；切回后会话空白 |
| **V1 逻辑保活** | `forceSwitchToLiveChat` 只做 `display` 切换；Immortal SSE 注册 | 与 `switchMainView('chat')` 混用，仍走同一容器；新建会话 `abortAllImmortalStreams()` 误杀技能流 |
| **V2 双轨容器（现行）** | **容器 A** `#chat-main-host`：普通闲聊 / 历史恢复；**容器 B** `#skill-live-host`：「正在使用的技能」专属活会话 | 视图切换仅 CSS 显隐；清理 / 新建只动 A；技能 SSE 只写 B |

核心状态变量（前端）：

- `_dualRouteActive`：`main` | `skill_live` — 当前可见主视图轨
- `_sseRenderHost`：`main` | `skill` — SSE / `appendMessage` 写入哪条 DOM 轨
- `_skillLiveSessionId` — 技能活会话 session，与 `currentSessionId`（主轨）解耦
- `registerImmortalController(..., { host: 'skill' })` — 标记不可被「新建会话」误 abort 的句柄

### 2. 病灶复盘

**高亮死锁**

- 旧版 `.sidebar-active-skill-wrapper.active-skill-entry` **默认带蓝色底色**，不区分「正在看技能会话」与「仅后台运行」。
- 切去技能广场后视图已变，但 CSS 仍保持「选中」外观 → 用户以为仍在前台会话。
- **修复**：底色改为仅 `.active-selected` 在 **技能活视图可见** 且存在 `_activeSkillPayload` 时启用；**呼吸灯** `.active-skill-running` 独立由 `setActiveSkillRunningState` 控制，切页不剥夺。

**误杀流**

- `clearChatAndReset` / `loadSessionAndRestore` 内调用 `abortAllImmortalStreams()` 或 `chatContent.innerHTML = ''`，把正在运行的技能 SSE 一并终止或丢失已渲染节点。
- **修复**：`abortMainChatStreamsOnly({ preserveSkill: true })`；清空范围限定 `#chatContent`；`#skillLiveContent` **禁止**在导航路由中清空或 abort。

**连续点击拦截**

- 捕获阶段委托在已处于 `#skill-live-host` 可见时仍重复触发 `forceSwitchToLiveChat`。
- **修复**：`isSkillLiveViewVisible()` 为真时 early-return，仅 `preventDefault` 不再重入。

### 3. 双轨路由入口（须遵守）

| 用户动作 | 入口 | 容器 |
|----------|------|------|
| 技能广场「使用 → 新建」 | `activateSkillLiveRoute` → `forceSwitchToLiveChat` | B |
| 侧栏「正在使用的技能」 | `forceSwitchToLiveChat` | B |
| 新建会话 / Logo 回首页 | `clearChatAndReset` / `clearMainChatOnly` | 仅清 A |
| 普通聊天 / 历史点进其他会话 | `switchMainView('chat')` / `loadSessionAndRestore` | A |
| 技能广场 / 历史 / 资产 | `switchMainView(...)` | 隐藏 A+B 中非目标 |

**禁止**：在 `switchMainView`、`showChatView`、`loadSessionAndRestore` 中对 `#skill-live-host` 执行 `innerHTML=''` 或 `abortAllImmortalStreams()`（除非用户显式「清除使用中技能」）。

### 4. 与时光机四联的关系

- 技能活会话的 **Live SSE** 仍须遵守上文四联：`thought` → `reasoning`，`message` → `text`，`status` → `process_log`。
- 历史恢复（`loadSessionAndRestore`）只恢复 **主轨** `#chatContent`；技能活会话若仍在 B 轨 DOM 中，同 sessionId 命中时走 `forceSwitchToLiveChat`，**不**重新 fetch 覆盖活 DOM。
- 变更 SSE 形态或快照写入时，除四联外须核对：双轨下 `getActiveChatContentEl()` / `.active-chat-content` 选择器是否仍指向正确容器。

### 5. 相关文件

| 路径 | 职责 |
|------|------|
| `services/nginx/html/index.html` | `#skill-live-host`、`forceSwitchToLiveChat`、`highlightActiveItemInSidebar`、`clearMainChatOnly` |
| `services/nginx/html/css/main.css` | `.active-selected` 与 `.active-skill-running` 分离 |
| `docs/持久化层构建规范.md` § 10 | 前端会话持久化与 DOM 隔离规范 |

## 🚨 [错题本/反面教材] 持久化过度工程引发的时序与 UI 坍塌

> **事件性质**：一次逻辑要求极强、最终以 **`git restore` 全量回滚至 `5fa6707`** 告终的底层重构（涉及 `step_rich_payload`、三段式 `workspace_outputs`、Markdown 霸权去重、规划期工作台 UI）。  
> **判定结论**：过度工程 —— 破坏实时 SSE 时序、引入前端互斥早退、引发 empty-state / CLI 株连污染。  
> **本文用途**：永久警示；后续 AI 协作与人类 review **必须先读本节** 再动持久化或历史渲染链路。

### 1. 背景与初衷 (The Intent)

历史「时光机」恢复工作台时，出现 **「吃数据」漏斗**：直播期间依次呈现 **数据诊断报告 → 执行结果详情（含图表/表格/Markdown）→ 专家分析报告**，但入库后再打开历史，常只剩最后一段专家报告；执行步骤仅有 `step_start` / `completed` 等状态壳子，**图表与富文本负载丢失**。

为修复该现象，曾尝试：

1. 引入 **`gibh_agent/core/step_rich_payload.py`**：在 Executor 步骤完成时构建 `rich_result_payload`（`rich_content` + `images` / `plot` / `tables` + `step_result_mirror` 深拷贝），绑定到 `steps_details[*]`，并扩展 `execution_snapshot.workspace_outputs` 三段式结构。
2. 前端增强 **`renderStepRichBody` / `hydratePureExecutionSteps`**：使历史恢复与直播工作台 1:1 对齐。
3. 为解决 **同一步骤图片叠影**（Markdown 内嵌图 + structured `images/plot` 双通道冗余），推演 **「Markdown 霸权 / 源头哈希去重」**：后端 SSOT 分支 + 前端互斥渲染。

上述改动在宿主机 pytest 与局部 DB 验证中「看似有效」，但集成到活会话后触发 **时序回退、图表消失、规划期全空白、CLI 输入框消失** 等连锁退化，最终整批撤销。

### 2. 致命的逻辑踩坑 (The Fatal Errors)

#### ❌ 时序破坏（SSE 延迟滥用）

**错误做法**：为「凑齐」持久化的三段式 JSON（诊断 / 执行 / 专家），在 `AgentOrchestrator` 中 **拦截本应在规划阶段（inspect 完成后）立刻 `yield diagnosis` 的实时 SSE**，改为执行全流程结束后再补发或静默 `_ensure_data_diagnosis_in_execution_snapshot` 才写入快照。

**恶果**：

- 直接违反 **流式交互「所见即所得」** 铁律：用户规划期看不到诊断报告，右侧工作台无法按设计自动展开。
- `diagnosis` 通道曾与 `expert_report` 混用（专家 Markdown 误走 `event: diagnosis`），导致 **`data_diagnosis_html` 入库为空** 而 `expert_report_markdown` 有值 —— 历史恢复更像「被覆盖」，实为 **通道错配 + 时序滞后** 叠加。
- 任何「等数据齐了再推给前端」的设计，本质是把 **持久化优先级置于 SSE 之上**，在组学 Agent 产品语义下不可接受。

#### ❌ 互斥渲染引发的数据黑洞（前端越权）

**错误做法**：为消除历史图片叠影，在前端 `renderStepRichBody` 引入 **互斥早退** —— 典型形态：`if (richMd) { 只渲染 Markdown; return; }` 或「有 `rich_content` 则跳过 `images` / `plot`」。

**恶果**：

- 前端 **越权替后端做 SSOT 裁决**：只要 Markdown 里有一句话，structured 图表负载（`images[]`、`plot`、`tables`）即被 **整段丢弃**。
- 历史步骤「有文字无图」—— 用户感知为 **历史记录变瞎**，比叠影更严重。
- 正确方向是 **后端去重、前端包容**：后端保证单一事实来源；前端 **顺序、无条件** 渲染各字段，禁止防御性截断。

#### ❌ DOM 株连与状态污染（CLI 输入框消失）

**错误做法**：

1. `appendMessage('user', …)` 内用 **`setTimeout(400ms)`** 延迟给 `#empty-state-wrapper` 加 `.hidden`，同时才 `appendChild` 将 `#inputWrapper` 迁入 `#input-wrapper-bottom`。
2. 双轨路由下 `#inputWrapper` 可能暂挂 **`#skill-input-wrapper-bottom`**（父容器 `#skill-live-host` 为 `display: none`），与 empty-state 隐藏时序叠加。
3. CSS `#chat-pane:has(#empty-state-wrapper:not(.hidden)) .chat-scroll-area { display: none }` 与空状态未同步退出竞态，曾引发 **规划期中栏全空白**。

**恶果**：

- 用户发送首条 prompt 后，**底部 CLI 整段消失**（`display: none` 株连或挂错隐藏父节点）。
- 「搬家 + 延迟隐藏」属于 **脆弱 imperative DOM**，与已确立的 **输入框物理独立于 empty-state** 结构相冲突。

**现行最小修复（基线 `5fa6707` + CLI patch）**：`dismissEmptyStateForChat()` —— **先**迁移输入框、**再**同步隐藏 empty-state；废除 `setTimeout`；`#input-wrapper-bottom` / `#inputWrapper` 加 `display: flex !important` 护航。**未恢复** rich payload / 互斥渲染 / 延迟 diagnosis 等实验逻辑。

### 3. 确立架构铁律 (Architectural Invariants)

基于上述血泪教训，凡改动 **SSE 形态**、**`state_snapshot` / `execution_snapshot` 写入**、**历史恢复渲染**、**empty-state / CLI 布局** 之一，须同时自检是否违反下列护栏。**绝对不可逾越**：

#### 🛡️ 法则一：实时流 (SSE) 绝对优先

- **诊断、工作流、步骤结果、status/thought/message** 等事件，须按业务时序 **立即** `yield` 给前端；用户感知延迟 ≈ 0。
- **持久化 / 入库** 只能是 **后台静默动作**（内存累加 → 回合结束或 checkpoint 统一 `save_message`），**严禁**为等 JSON 结构完整而 **阻塞、拦截、推迟** SSE。
- **禁止**把专家报告、执行摘要等写入 `event: diagnosis` 等 **语义错误的通道**；字段与事件类型须与 `docs/持久化层构建规范.md`、本节四联表一致。

#### 🛡️ 法则二：源头 SSOT 去重 + 前端包容性渲染

- **所有去重**（路径/base64 哈希、Markdown 内嵌图 vs `images/plot` 互斥、表格重复等）**必须在后端**（如 `step_rich_payload` 构建阶段）完成，并写入 **单一 canonical 字段集**。
- 前端是 **哑终端（Dumb Terminal）**：`renderStepRichBody` 等须 **单通道、包容式、顺序渲染** —— `rich_content` → `images` / `plot` / `tables` 各段 **能 render 就 render**，**禁止** `if (rich_content) return` 类互斥早退。
- 若仍出现叠影，**回后端查冗余 payload**，不得再在前端「聪明地藏图」。

#### 🛡️ 法则三：物理级隔离与 CSS 霸权

- **CLI 输入框**（`#inputWrapper` / `#input-wrapper-bottom`）与 **空状态**（`#empty-state-wrapper`）、**双轨聊天主容器**（`#chat-main-host` / `#skill-live-host`）在 DOM 上 **兄弟或独立槽位**，**禁止**将输入框嵌套在 empty-state 内。
- 空状态退出：**同步** `dismissEmptyStateForChat()`（先迁输入框 → 再 `.hidden`），**禁止** `setTimeout` + 跨容器 `appendChild` 反复搬家。
- 核心交互容器须有 **CSS 可见性霸权**（如 `display: flex !important; visibility: visible !important; z-index: 1000 !important`），避免 `:has()` / 父级 `display:none` 株连。
- 双轨切换时 `moveComposerInputToHost('main'|'skill')` 须与 **`isSkillLiveViewVisible()`** 一致；主轨可见时 **不得**把输入框留在隐藏的 skill 槽位。
