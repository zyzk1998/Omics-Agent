# 技能扩展规范文档（AI 编程助手 SOP）

> **受众**：负责技能扩容的 AI 编程助手与研发人员。  
> **目标**：任何新技能接入遵循同一套步骤，避免破坏语义路由、工具注册表与前端技能广场布局。

**索引快照（与种子数据同步刷新）：** 修改 `gibh_agent/db/seed_skills.py` 后，在仓库根执行 `PYTHONPATH=. python3 scripts/export_skill_library_md.py` 更新根目录 `[技能库.md](../技能库.md)`（`name` / 主副类 / 简介 / `skill_route_tool`）；**不含**动态插件与仅 DB 存在的条目。

## 1. 技能元数据注册（前端 & 数据库）

### 1.1 内置/系统技能（`skills` 表）

模型定义见 `gibh_agent/db/models.py` 中 `Skill`：

- `**name`**：全局唯一标识（英文 slug 为佳），系统预置技能勿与用户冲突。
- `**description`**：卡片摘要，展示在技能广场。
- `**main_category` / `sub_category`**：筛选与标签；前端 `index.html` 中 `skillCardIcons` 常按 `**sub_category`** 映射图标（缺省则回退默认图标）。
- `**prompt_template**`：用户从广场发起会话时注入的模板。  
  - 若需 **路由快车道**，须与编排器约定格式（例如含 `[Skill_Route: ...]` / `[Omics_Route: ...]` 等），见种子数据 `gibh_agent/db/seed_skills.py`。
- `**author_id`、`status`**：上架需 `status="approved"`。

列表接口：`GET /api/skills`（`server.py` 与 `gibh_agent/api/routers/skills.py` 同源逻辑），返回字段驱动前端卡片：**名称、描述、主/子类、`prompt_template`、收藏状态等**。

### 1.2 动态插件技能（ZIP 上传 → `dynamic_skill_plugins`）

适用于「用户上传包」路径，注册逻辑见 `gibh_agent/plugin_system/registry.py`：

- 持久化字段含 `**display_name`、`description`、`parameters_schema`（JSON Schema）**、`**skill_type`**（`prompt` | `script`）、`**script_path**`（script 类型必填，指向解压后 `main.py` 绝对路径）、`**extract_dir**`、`**worker_route**`。
- 广场展示通过 `plugin_to_skill_plaza_payload()` 转成与 `Skill` 列表兼容的 JSON；`**id` 形如 `dyn_{数字}**`，前端需识别 `**is_dynamic_plugin**`（见 `server.py` 合并列表逻辑）。

参数表单：**由 `parameters_schema` 定义**，解析入口见 `gibh_agent/plugin_system/parser.py`（从 Markdown ZIP 元数据生成 OpenAI 风格 schema）。

### 1.3 前端「技能广场」渲染要点（避免破坏布局）

文件：`services/nginx/html/index.html`。

- 卡片结构为 **Flex 布局**，新增样式时勿破坏现有 flex 与栅格。
- **图标**：`skillCardIcons[sub_category]`；无映射时使用默认（如 🧬）。新增子类时应 **同步增加映射或接受默认图标**。
- **纯文档类技能**：后端可标记 `plain_doc_skill`（若当前接口已支持），前端使用中性图标与 `skill-card--research-doc` 样式，避免与普通技能混淆。
- **动态插件**：卡片上已有 `is_dynamic_plugin` 等行为分支；新逻辑尽量走数据字段而非硬编码 id。

 UTF-8、微创修改：禁止全量替换 `index.html`。

## 2. 后端工具类挂载（`gibh_agent/tools/` & Tool Registry）

### 2.1 加载机制

- `gibh_agent/tools/__init__.py` 中 `**load_all_tools()`** 递归导入包内模块，触发装饰器注册。
- 新增工具必须能被该导入链加载（例如在已有子模块中导出，或确保包内可被 `pkgutil.walk_packages` 发现）。

### 2.2 注册方式（非「继承类」，而是装饰器注册）

核心：`gibh_agent/core/tool_registry.py` 中 `**ToolRegistry.register`**。

- 使用 `**@registry.register(name="...", description="...", category="...")`** 装饰异步或同步函数。
- **参数 Schema**：由函数签名与类型注解 **自动生成 Pydantic 模型**（无需手写类继承 `Tool`，除非团队另有封装）。
- `**category`**：影响 `capabilities_digest` 中的分组展示；命名宜稳定、可读。

### 2.3 返回值与安全执行

- 工具函数应使用 `**@safe_tool_execution`**（`gibh_agent/core/utils.py`），统一 `**{"status": "success"|"error", "message": "...", ...}`**。
- **路径类参数命名**必须遵守 `.cursor/rules/gibh-architecture-constitution.mdc` 中的词汇表（如 `file_path`、`input_dir`、`sequence_or_path` 等），**禁止自造路径参数名**，以免诊断注入与反射执行错位。

### 2.4 重型计算

严禁在智能体主进程跑重 CPU/GPU；应 **独立 Worker / TaaS 微服务** 或 `**asyncio.create_subprocess_exec`** 隔离（见架构宪法）。

## 3. 路由与大模型视野（Semantic Router & Clarify）

### 3.1 Semantic Router 的动态「工具视野」

文件：`gibh_agent/core/semantic_router.py`。

- 运行时 `**build_semantic_router_system_prompt()`** = 静态红线 + **【F 段】动态清单**。
- 【F 段】内容由 `**gibh_agent/core/capabilities_digest.py`** 的 `**build_capabilities_digest_for_llm()`** 生成：
  - **SOP**：`WorkflowRegistry.list_workflows()` 的域名与描述摘要；
  - **工具**：`ToolRegistry` 中已注册工具，按 `**category`** 聚合 `**tool_id`**（有数量与长度上限，过长会截断）。

因此：**想让模型在路由阶段「知道」某项能力**，通常必须满足至少其一：

1. **注册对应原子工具**（并被 `load_all_tools()` 加载）；或
2. **注册对应工作流到 WorkflowRegistry**；或
3. **在极少数场景**扩展 `SEMANTIC_ROUTER_SYSTEM_PROMPT_BASE` 中的硬性规则（需谨慎，避免与 digest 矛盾）。

数据库里仅有一条 **Prompt 技能**但没有对应工具/工作流注册时，模型在 **F 段可能看不到**，除非通过模板或编排其它环节注入。

### 3.2 `skill_fast_lane`（快车道）

语义路由枚举见 `RouteKind.skill_fast_lane`。当前基座 Prompt 内含 **示例性** 成药/化学工具规则（如 `**skill_id=drug_similarity`**）。

- **新增快车道技能**不仅是加 DB 记录，通常还需：**编排器识别 `skill_id`、与 `prompt_template` 对齐、必要时更新 `SEMANTIC_ROUTER_SYSTEM_PROMPT_BASE` 与评测用例**（参考 `eval_semantic_router.py`）。
- 大多数组学能力应走 `**task` + SOP/工具**，不必强行快车道。

### 3.3 Clarify / 澄清模式与 capabilities digest

- **语义路由失败进入「澄清型聊天」时**：`AgentOrchestrator` 会在 system prompt 中注入 `**build_capabilities_digest_for_llm()`**，用于告知用户**真实可用的 SOP 与工具**，并配合「⚡ 启动工作流：」等约定（见 `orchestrator.py` 中 `semantic_clarify_rationale` 分支）。
- `gibh_agent/core/agentic.py` 中的 `**Clarifier`** 类侧重执行前业务澄清，其内置英文模板**不一定**携带完整 digest；与「路由澄清聊天」路径不同。扩容时以 **orchestrator 注入 digest** 的路径为准。

## 4. 依赖与资产管理（Assets）

### 4.1 统一目录布局（服务器磁盘）

技能相关底层脚本、文档与模型权重，统一放在 `**${UPLOAD_DIR}/skills_assets`** 下（可用环境变量 `**SKILLS_ASSETS_ROOT`** 整体覆盖）。默认 `**UPLOAD_DIR=/app/uploads`**（Docker 与 `docker-compose.yml` 中 `./data/uploads:/app/uploads` 挂载一致）。


| 分类                | 路径（容器内）                                | 用途                                                                            |
| ----------------- | -------------------------------------- | ----------------------------------------------------------------------------- |
| **纯 Prompt / 文档** | `{SKILLS_ASSETS_ROOT}/prompt/`         | SKILL.md、说明文档、上架前校验材料                                                         |
| **脚本类**           | `{SKILLS_ASSETS_ROOT}/script/inbound/` | 指挥上传的 **zip 暂存**（入库前检查）                                                       |
| **脚本类（运行时）**      | `{SKILLS_ASSETS_ROOT}/script/runtime/` | ZIP **安全解压**目录（`uuid/` 子目录）、`main.py` 沙箱根；**即动态技能的 `DYNAMIC_SKILLS_DIR` 默认值** |
| **模型权重**          | `{SKILLS_ASSETS_ROOT}/models/`         | 大体积权重、二进制；由工具参数或配置引用路径                                                        |


实现与 API：`gibh_agent/core/skills_assets_layout.py`。`gibh_agent/plugin_system/parser.py` 解压目标为 `**get_dynamic_skills_dir()`**（优先环境变量 `**DYNAMIC_SKILLS_DIR`**，否则为 `**script/runtime**`）。Worker 侧 `workers/worker-pyskills/dynamic_runner.py` 使用**相同规则**解析沙箱根目录（镜像内不含 `gibh_agent`，逻辑内联保持一致）。

**宿主机典型路径（本仓库 docker-compose）**：

```text
<仓库根>/data/uploads/skills_assets/prompt/
<仓库根>/data/uploads/skills_assets/script/inbound/
<仓库根>/data/uploads/skills_assets/script/runtime/
<仓库根>/data/uploads/skills_assets/models/
```

首次一键创建（在服务器上、于仓库根目录执行）：

```bash
mkdir -p data/uploads/skills_assets/{prompt,models,script/inbound,script/runtime}
```

**升级说明（旧目录名 `dynamic_skills`）**：若线上历史数据 `extract_dir` 仍指向 `**/app/uploads/dynamic_skills`**，在未迁移文件前可继续设置 `**DYNAMIC_SKILLS_DIR=/app/uploads/dynamic_skills`**，或完成数据迁移后再改用新默认路径。

### 4.2 脚本与运行时依赖

- **随仓库发布的轻量脚本**：放在 `**gibh_agent/assets/<领域>/`**（随 `COPY gibh_agent/` 进入镜像）、或仓库根 `scripts/` 等稳定路径；由工具内 `**subprocess`** 调用，路径通过 `**file_path`/`input_dir**` 传入用户数据。
- **大型 conda/R 环境**：放在镜像或 Worker 中，通过环境变量指向解释器；**勿在已跟踪文件中写真实内网地址或密钥**（见 `.cursor/rules/git-no-secrets-in-tracked-files.mdc`）。

### 4.3 内置脚本类原子工具（实践范式 · EIS/DRT）

以下自 **电化学 DRT（`eis_drt_analysis`）** 打通全流程后的经验，可作为同类「工具注册 + 子进程脚本 + 快车道」技能的模板。


| 环节                            | 做法                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **注册与加载**                     | 在 `gibh_agent/tools/` 下实现工具，`@registry.register(name="...", ...)` 的 `**name`** 与 `**prompt_template` 首行 `[Skill_Route: …]`** 完全一致；确保模块被 `load_all_tools()` 导入。                                                                                                                                                                                                                                                                                  |
| **底层脚本位置**                    | 推荐随仓库放置 `**gibh_agent/assets/<名称>/`**（例：`gibh_agent/assets/drt/drt_analysis.py`），Dockerfile 已 `**COPY gibh_agent/`**，无需依赖手工解压 ZIP。可选环境变量 `**DRT_ANALYSIS_SCRIPT**`（或领域等价变量）覆盖为外部绝对路径。                                                                                                                                                                                                                                                           |
| **隔离执行**                      | 禁止在主进程内 import 重型第三方脚本栈；使用 `**subprocess.run`**（或等价）调用脚本，超时与 `**capture_output`** 可观测排障。                                                                                                                                                                                                                                                                                                                                                        |
| **CSV/数据兼容**                  | 脚本侧应对 **多编码、分隔符、表头别名**（如 `Freq/Hz`、`Z'/ohm`、`Z''/ohm`）做解析；数值病态时优先 `**lsq_linear`/`NNLS`** 等有界或稳健求解，避免裸 `lstsq` 导致失败。                                                                                                                                                                                                                                                                                                                            |
| **编排快车道**                     | 用户消息含 `**[Skill_Route: <tool_id>]`** 时，`AgentOrchestrator` 早于全局意图分流，交由 `**SkillAgent.execute_skill`** → `**registry.get_tool(tool_id)`**；**无需**为每个新工具改编排器分支（与语义路由 `skill_fast_lane` 补全暗号的路径不同）。                                                                                                                                                                                                                                                 |
| **存量数据库**                     | 系统技能仅在 `**skills` 表 `COUNT(*)==0`** 时 `**run_upsert_system_skills`** 补种；线上已有库不会自动刷新模板。若需补 `**[Skill_Route: …]`** 等字段，使用仓库内 **一次性 UPDATE 脚本**（示例：`scripts/patch_drt_skill.py`），**不要用「行数阈值反复补种」**。                                                                                                                                                                                                                                                |
| **前端右栏（`skill_tool_result`）** | 工具 `**status: success`** 时，除 `**data`** 外建议返回：`**markdown`**（含同源图：可用 `**![](/results/...)`** 或 与 DRT 一致的带样式 HTML `<img>` 容器）、`**image_urls**`（`/results/` 开头的同源 URL 列表）、`**json_url`**（可选，指向 `drt_summary.json` / `chem_summary.json` 等）。仅返回 `**data.image_paths` 磁盘绝对路径**时，右侧 `**buildSkillToolWorkspace`** **无法**可靠出图（历史问题已修复范式）。映射规则：将 `**RESULTS_DIR`/`UPLOAD_DIR`** 下的绝对路径转为 `**/results/...`、`/uploads/...`**（与 `services/nginx` 静态别名一致）。 |
| **前端微创**                      | `index.html` 中明细区 `**gridSkip`** 宜跳过 `**image_paths`、`json_paths`、`output_dir`** 等长字段，避免撑爆指标栅格；脚注可增加 **「下载 JSON」**（`json_url`）与 CSV/ZIP 并列。                                                                                                                                                                                                                                                                                                     |


**验收**：同路径下 pytest 冒烟（例：`tests/test_drt_analysis_smoke.py`）或手工上传技能包中的测试 CSV，确认子进程退出码 0、右栏 Markdown 与图片可见。

### 4.4 化学 RDKit 批量子工具（与 §4.3 / DRT 对齐）

以下内容与 `**eis_drt_analysis`** 的系统融合方式一致，便于扩展与审计对照。


| 项目                     | 说明                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **脚本入口**               | `**gibh_agent/assets/chem_rdkit/run_chem_tool.py`**（子进程）；可选 `**CHEM_RDKIT_RUNNER`** 覆盖脚本路径。解释器 `**CHEM_RDKIT_PYTHON**`：`chem_rdkit_tools` 优先读取该环境变量，**勿默认假设** `sys.executable` 已装 RDKit。                                                                                                                                                                                                                                                                                                                                    |
| **Docker / 部署硬约束**     | `**services/api/Dockerfile`**：① 系统包 含 `**libxrender1`、 `libxext6`、 `libfontconfig1`、 `libxft2`、 `libcairo2`**（RDKit `**Draw` / `MolToImage**` 依赖 `**libXrender.so.1**` 等，slim 镜像默认无）。② `**/opt/chem-rdkit-venv**` 内先装 `**numpy>=1.24,<2**` 再装 `**rdkit-pypi**`，避免 **NumPy 2.x** 与 RDKit 扩展的 `**_ARRAY_API` / `rdInfoTheory` 崩溃**（与主 `requirements.txt` 的 `numpy<2` 一致）。构建阶段执行 `**from rdkit.Chem import AllChem, Descriptors, Draw`** 失败则镜像失败。`**ENV CHEM_RDKIT_PYTHON=...`** 与 **compose** 注入同前；**entrypoint** 用相同学步长幂等重建。 |
| **容器内验收（防再犯）**         | `docker exec gibh_v2_api /opt/chem-rdkit-venv/bin/python -c "import numpy as np; print(np.__version__); from rdkit.Chem import Draw; print('draw ok')"`（应 **numpy 1.x**）；再任选 `**chem_molecular_weight`** 跑技能确认 exit 0。                                                                                                                                                                                                                                                                                                      |
| **原子工具模块**             | `**gibh_agent/tools/chem_rdkit_tools.py`**，注册 `**chem_bbb_assessment`、`chem_pains_filter`、`chem_brenk_filter`、`chem_morgan_fingerprint`、`chem_molecular_weight`、`chem_tanimoto_similarity`**；由 `**tools/__init__.py**` 显式加载（与 `**drt_tool**` 并列）。                                                                                                                                                                                                                                                                           |
| `**[Skill_Route: …]**` | `**seed_skills.py**` 对应技能 `**prompt_template` 首行**与各 `**@registry.register(name=...)`** 一致；存量库用 `**scripts/patch_chem_skills_batch.py**`（与 `**patch_drt_skill.py**` 同理）。                                                                                                                                                                                                                                                                                                                                                    |

## 5. 前端表现层与上线规范（Frontend & Launch SOP）

### 5.1 数据从哪里来

- **主路径**：前端 `services/nginx/html/index.html` 中 `**fetchSkills`** 调用 `**GET /api/skills`**，列表项字段来自 `**gibh_agent/db/models.py` · `Skill` 表**（及少量「特色科研流程」前端静态 Featured 数组）。
- **补种**：空库或按需触发 `**gibh_agent/db/seed_skills.py`** 的 `**run_upsert_system_skills`**，统一写入系统技能。

### 5.2 必备字段与 `is_implemented`（已实现置顶 · 不落库）


| 字段                                   | 含义                                                                                                                                                                                |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `**main_category` / `sub_category`** | 大类/细分品类；决定筛选 Tab 与 `**skillCardIcons[sub_category]`** 映射（见 `index.html`）。                                                                                                         |
| `**description`**                    | 卡片摘要；建议 **≤220 字等效展示**（前端按约 220 字符截断并加省略号，避免撑破卡片）。                                                                                                                                |
| `**prompt_template`**                | 用户点「使用 → 新建会话」时写入输入框的全文；须含 `**[Skill_Route: 工具ID]`**（与 `**@registry.register(name=...)`** 完全一致）方可进入编排快车道。                                                                         |
| `**is_implemented**`（仅 JSON）         | **不在数据库持久化**。由 `**gibh_agent/core/skill_plaza_utils.infer_skill_implemented_from_prompt`** 根据模板内是否含有 `**[Skill_Route:`** / `**[Omics_Route:`** 动态计算并附加到 `**GET /api/skills`** 响应。 |


**置顶原理（双保险）**

1. **API**：对筛选后的 ORM 结果在 **Python 内存**中排序：**「含快车道暗号」优先**，其次 `**created_at` 降序**。**不使用** SQL `ORDER BY` 依赖任何「已实现」物理列。
2. **前端**：在合并 Featured 精选**之前**，对 API 返回的 `**items`** 调用 `**sortSkillsPinImplemented`**（定义于 `**fetchSkills` 上方的独立函数**），保证「已接入」排在「规划中」前；**同为「已接入」时按 `created_at` 降序**，与 API 一致。Featured 精选仍合并逻辑不变（如「全部」首页仍可出现精选置顶区）。

融合收尾时 **刷新 `created_at`** 使新技能在同档内置顶 — 见 **§5.8**。

动态插件：`**plugin_to_skill_plaza_payload`** 对 `**prompt`** 正文调用同一推断函数设置 `**is_implemented`**。

### 5.3 Prompt 模板与 `[Skill_Route: xxx]` 写法

参考 `**seed_skills.py`** 文件头注释（五段式规范）。推荐结构：

1. **首行**：`[Skill_Route: <tool_id>]`（若需 Omics 域亦可保留 `[Omics_Route: …]` 约定，与编排器一致即可）。
2. **一两句用户开场**，说明科学目标与交付形态。
3. **参数/数据说明**：列出工具参数名与生物学含义；**明确写出「请在此替换原始序列 / 上传数据文件」**等占位提示。
4. **文末「助手侧」**：提示 LLM 如何将附件路径写入 `**file_path`** / `**sequence_or_path`** 等（遵守路径参数词汇表）。

**化学 / 电化学示例**：`**DRTtools`** — `sub_category` **「电化学」**，模板首行 `**[Skill_Route: eis_drt_analysis]`**，与 `**gibh_agent/tools/drt_tool.py`** 注册名一致。**药物化学 RDKit** — `sub_category` **「药物筛选」**或 「分子计算」，模板首行 `**[Skill_Route: chem_*]`**，与 `**gibh_agent/tools/chem_rdkit_tools.py**` 注册名一致（参见 §4.4）。

### 5.4 前端卡片一致性（Icons / Tags）

- **图标**：在 `**index.html`** 的 `**skillCardIcons`** 中为新品类增加 emoji 映射（如 `**'电化学': '🔋'`**、`**'药物筛选': '💊'`**、`**'分子计算': '⚗️'**`）；勿破坏现有 Flex 卡片结构。
- **标签**：同一渲染分支内使用 **「动态工作流·已接入」/「动态工作流·规划中」** + **子类标签**，与 `**is_implemented`** 联动。

### 5.5 上线检查清单

- 空库时 `**GET /api/skills`** 会触发 `**total==0`** 路径下的 `**run_upsert_system_skills`**（**禁止**「行数不够就反复补种」等非空表策略）。
- `**prompt_template`** 含正确 `**Skill_Route`**，且工具已在 `**ToolRegistry`** 注册。
- **非空库上线新快车道技能**：种子不会更新已有行时，执行 **一次性 SQL/脚本 PATCH**（如 `**scripts/patch_drt_skill.py`**）写入暗号与分类，或采用等价受控迁移。
- 前端 `**fetchSkills`** 能收到 `**is_implemented`**（计算字段）；卡片描述长度与图标正常。
- 若需在右栏展示运算结果：工具成功载荷含 `**markdown` + `image_urls`**（及可选 `**json_url`**），URL 为 `**/results/**` 等同源路径而非仅磁盘绝对路径（RDKit 六工具与 DRT 同构，见 §4.4）。
- 自主测试通过后执行 **§5.8 置顶收尾**（PATCH 刷新 `created_at` 或等价 SQL）。

### 5.6 向总指挥汇报规范（交付铁律）

- **名称 ↔ ID 映射表为强制交付物**：凡「技能融合 / 快车道接入」类任务结项时，AI 或研发在总结报告、MR 描述、验收邮件中 **必须** 提供表格：**【前端技能广场卡片名称（`Skill.name`）→ 底层 Tool ID（`@registry.register` 的 `name`）】**。  
- **严禁**仅用底层 ID（如 `chem_molecular_weight`）汇报而无卡片名称——总指挥与产品侧无法仅凭 ID 对齐广场 UI 与验收范围。  
- RDKit 六工具映射见 **§4.4** 表格；其它技能沿用同一格式自行扩展。

### 5.7 数据库 / Seed 中配置含 `[Skill_Route: …]` 的交互 Prompt

快车道依赖编排器解析用户消息中的 `**[Skill_Route: <tool_id>]`**，且 `**tool_id`** 必须与 `**ToolRegistry**` 注册名 **逐字一致**。推荐做法：

1. **单一来源**：将完整 `**prompt_template`** 集中在 `**gibh_agent/db/chem_skill_prompt_templates.py`**（化学六工具示例），由 `**seed_skills.py`** 在合并 `ADDITIONAL_CHEMISTRY_SKILLS` 后注入，`**scripts/patch_chem_skills_batch.py**` 从同一字典读取，避免 Seed 与 PATCH 分叉。
2. **结构**：**首行**必须是 `**[Skill_Route: …]`**（不加前导空行）；紧随业务引导语；**必须**包含对用户可见的 **SMILES / 文件输入说明**（例如「请在此输入 SMILES 字符串，或上传包含 SMILES 的 `.csv`/`.txt` 文件：`___`」），与 §5.3 五段式兼容。
3. **存量库**：仅空库自动补种会写入 Seed；线上须执行 **一次性 PATCH 脚本** 更新 `prompt_template`（见 §4.3「存量数据库」）。
4. **验收**：`GET /api/skills` 中 `**is_implemented`** 为真（模板内含暗号）；新建会话注入全文后编排 **SkillAgent** 可走快车道。

### 5.8 融合收尾：技能广场置顶（自主测试通过后的最后一步）

技能广场列表在 `**GET /api/skills`**（`gibh_agent/api/routers/skills.py` 与 `**server.py`** 同源逻辑）中的稳定排序为：

1. `**is_implemented**` 为真（含 `**[Skill_Route:**` / `**[Omics_Route:**`）的条目 **整体靠前**；
2. **同档内**按 `**created_at` 降序**（越新越靠前）。

因此：**自测通过、确认可上线后**，应对新接入/刚 PATCH 的技能行刷新 `**skills.created_at`**（UTC），使其在「已接入」分组内排到最前，用户打开「**全部**」或「**化学**」等 Tab 时无需翻页即可看到最新融合项。

**推荐做法**

- **化学 RDKit 六工具存量库**：执行 `**scripts/patch_chem_skills_batch.py`** — 默认在 UPDATE 时 **同步将 `created_at` 设为当前 UTC**；若仅改文案不希望动排序，附加 `**--no-touch-created-at`**。
- **其它技能**：在一次性 PATCH 脚本或受控迁移中，对目标行执行 `**UPDATE ... SET created_at = UTC_TIMESTAMP() ...`**（或与 ORM 等价的 `datetime.utcnow()`），原则与上相同。
- **前端双保险**：`services/nginx/html/index.html` 中 `**sortSkillsPinImplemented`** 在同为「已接入」时按 `**created_at`** 再排一次，与 API 语义一致。

**勿与「精选 Featured」混淆**：`**window.__FEATURED_RESEARCH_SKILLS`** 仅用于少量橱窗位，且常带 **非数字 `feat_*` id** — **收藏依赖真实 `skills.id`**。常规新技能置顶应走 `**created_at` + 快车道暗号**，不要为「抢首屏」大量塞 Featured 假卡片。
