# Omics Agent 身份认证体系（架构宪法）

> 本文档为**设计与治理基线**，用于约束后续实现与评审。与当前代码不一致之处，以本文档为**目标态**；落地时需配套迁移计划与兼容性说明。

---

## 1. 认证核心基建（Credential & Crypto Baseline）

### 1.1 访问令牌：JWT

- **主凭证形态**：采用 **JSON Web Token（JWT）** 作为跨端访问令牌，经 **HTTPS** 传输，服务端校验签名与有效期。
- **算法与密钥**：对称或非对称算法由部署配置锁定；**密钥轮换**须纳入运维流程，禁止在仓库或镜像中硬编码生产密钥。
- **有效期策略**：
  - **Access Token**：短期有效；目标态默认 **7 天**（与现行部署习惯对齐，可按环境下调）。
  - **无感刷新**：**待定**。在未引入 Refresh Token / 轮转机制前，产品侧须接受「到期重新登录」或「静默续期窗口」的显式取舍；一旦引入 Refresh，须规定：存储位置（建议 HttpOnly Cookie 或安全区）、吊销与盗用检测。
- **载荷最小化**：JWT 仅承载身份解析所需声明（如 `sub`、角色、签发时间等），**禁止**放入 PII 明文或可逆敏感信息。

### 1.2 密码存储：强哈希（Bcrypt）

- **算法**：密码入库前必须经过 **bcrypt**（或经安全评审批准的等价慢哈希）处理；与现有 `pwdlib`/`BcryptHasher` 路线一致。
- **策略**：每密码独立盐、工作因子可配置；**禁止**明文或可逆加密存放密码。
- **传输**：仅允许在 TLS 保护下的登录/MFA 通道提交口令；日志与 APM **脱敏**。

## 2. 注册与登录流转规范（Identity Lifecycle）

### 2.1 注册硬性约束

- **用户名**：全局唯一、不可变或可变更须走风控审计（目标态：**注册时必须提供唯一用户名**）。
- **触达渠道（强制）**：须至少绑定 **邮箱或手机号其一**（推荐二者择一强制 + 另一可选），用于 **MFA、通知、找回与风控**；未验证的渠道不得视为可信触达。
- **验证**：注册流须包含 **验证码或等价证明**（邮件/短信），以确认渠道归属；**禁止**仅凭自填邮箱/手机即标记为已验证。

### 2.2 三合一登录入口（Identifier Unification）

- **单一主输入**：登录第一步仅展示 **一个聚合输入框**，占位提示为：`请输入用户名 / 邮箱 / 手机号`。
- **自动分流**：服务端（或受信网关）根据语法与目录查询判定标识类型，路由至统一身份解析管线；**禁止**要求用户自行选择「用哪种方式登录」作为前置步骤（可作为辅助纠错入口）。
- **错误与防枚举**：对外错误文案需统一（如「账号或密码错误」），详细原因仅入审计日志。

### 2.3 MFA（二次验证）机制

- **强制顺序**：**口令校验通过后，必须进入第二因子验证**（邮箱 OTP / SMS OTP / 未来可扩展 TOTP 硬件令牌等）。
- **通道选择**：若用户同时绑定多种触达，须允许在策略允许范围内选择发送通道；默认策略由安全与产品共同定义。
- **票据与防重放**：MFA 阶段使用 **短时、单次消费** 的验证票据（或等价会话态），与最终签发的 **Access JWT** 解耦；验证失败次数与冷却须有限流。

## 3. OAuth 关联与防劫持（Ownership Challenge）

### 3.1 原则：拒绝静默合并

- **禁止**在未经本地身份确认的情况下，将第三方账号（如 GitHub、微信、Google）与已有本地账户 **静默绑定或自动合并**。
- **首次关联**：须在用户已登录态下发起绑定，或经 **所有权挑战** 后方可完成关联。

### 3.2 邮箱冲突与所有权挑战

- 当第三方 IdP 返回的 **邮箱** 与本地已注册邮箱 **冲突** 时：
  - **不得**直接写入绑定关系；
  - 必须进入 **所有权挑战** 流程：要求用户输入 **本地账户密码**（或已启用的更强验证），校验通过后方可完成「GitHub ↔ 本地账号」绑定。
- **审计**：绑定/解绑操作须记录操作者、时间、IdP、`sub`、IP 与设备指纹（若可用）。

## 4. 客户端沙盒与提权模型（Desktop IPC Privilege）

### 4.1 权限隔离总原则

- **Web 前端（含 Electron 内嵌 Renderer 中的网页）**：视为 **不可信 UI**，**严禁**直接调用 **Node.js API**、文件系统、子进程或任意 **高权限 IPC**。
- **Electron 主进程 / Preload 特权层**：唯一可信边界；默认 **拒绝** 所有特权操作。

### 4.2 提权：短期 Token 解锁 IPC

- **模型**：本地敏感能力（如打开终端、执行受控命令、读写受保护路径）仅在 **主进程鉴权通过** 后，向 **隔离上下文** 下发 **短期、窄权限** 的 **操作令牌**（非用户长期 JWT，可与用户 Access JWT 分域）。
- **范围**：令牌须绑定 **操作类型、资源范围、时间窗、单次或有限次数**；过期立即失效。
- **通道**：Renderer 仅通过 **受控 IPC 契约**（如 `invoke('run-approved-task', payload + opToken)`）发起请求；主进程校验令牌与参数白名单后再执行。
- **迁移要求**：现有 `nodeIntegration: true` / `contextIsolation: false` 等配置与目标态 **不兼容**，须在桌面端硬化路线中列为 **必须整改项**。

## 5. 与现有实现的差距说明（Gap，供总指挥排期）

| 域 | 现状（摘要） | 目标态（本文） |
|----|--------------|----------------|
| Token | 单 JWT、无 Refresh | Access 短期 + Refresh 待定；MFA 票据独立 |
| 前端存储 | localStorage Bearer | 迁移至 HttpOnly / 分区存储等硬化方案待设计 |
| 注册 | 用户名+密码 | + 强制触达 + 验证 |
| 登录 | 用户名+密码直出 JWT | 三合一标识 + MFA 强制 |
| OAuth | UI 占位 | 显式绑定 + 所有权挑战 |
| 桌面 | 薄壳加载 Web + 高风险的 webPreferences | 沙盒 + IPC 令牌提权 |
