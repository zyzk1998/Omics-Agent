# 底层算子融合智能体系统标准方法论

## 1. 架构认知：三层汉堡模型

本系统将智能体与算子解耦为三层，自上而下依次为：

| 层级 | 比喻 | 职责 | 典型代码位置 |
|------|------|------|----------------|
| **L1 大模型** | 大脑 | 理解用户意图、在快车道场景下**抽取结构化 JSON 参数**（或 function call），**不执行**科学计算 | `LLMClient`、`stream_from_llm_chunks`、`SkillAgent._messages_for_streaming_arg_extract` |
| **L2 智能体适配器** | 神经—肌肉接头 | **唯一允许**调用 `ToolRegistry` 已注册工具；负责路径校验、子进程/线程隔离、超时、错误归一化为 `dict`；**禁止**在此层直接 `import` 巨型 DL 栈并占满 GPU（除非经架构评审） | `SkillAgent.execute_skill`、`gibh_agent/tools/*.py` 中 `@registry.register` 的异步函数 |
| **L3 底层算子** | 肌肉 | **黑盒**：CLI / 独立 Python 脚本 / 容器内服务；完成数值计算与文件产出；**不知道** FastAPI、SSE、Skill_Route | 运维部署目录如 `/opt/pipelines/...`，由适配器通过**参数与文件路径**驱动 |

### 1.1 数据与路径：OmicsAssetManager（资产总线）

上传与多文件场景下，Orchestrator / Executor 使用 **`OmicsAssetManager`**（`gibh_agent/core/asset_manager.py`）将路径规范为 `DataAsset` 列表，并向下游暴露 **`fasta`** 等桶及解析后的绝对路径。

**新人工作边界**：

- **算子层**：只接受「已解析的绝对路径」或标准输入格式；**不**自行猜测上传目录结构。  
- **适配器层**：从 `SkillAgent` 传入的 `file_paths` 或工具参数中选择合法路径传入算子；**禁止编造路径**（与 `SkillAgent` 系统提示一致）。  
- **不得**在算子内写死服务器私有绝对路径（见第 2 节）。

### 1.2 快车道入口：Orchestrator 硬拦截

`Orchestrator` 对用户查询做正则匹配：`[Skill_Route: tool_name]`（`tool_name` 必须为 `[a-zA-Z0-9_]+`），命中后**绕开**全局意图分类与 SOPPlanner，直接进入 **`SkillAgent.execute_skill`**。

**结论**：新人的交付必须保证 **`tool_name` 与 `ToolRegistry` 中 `registry.register(name=...)` 完全一致**，否则快车道无法调用。

## 2. 底层算子交付标准（黑盒原则）

以下条款为**验收必要条件**；不满足者不得合并入主分支。

### 2.1 接口形态

1. **必须**支持 **命令行参数**（`argparse` 或等价），或 **单一入口 Python 模块**（`python -m pkg.run --help` 可打印用法）。  
2. **必须**在失败时以 **非零 exit code** 退出，并向 **stderr** 输出人类可读错误信息（stdout 可用于 JSON 结果或保持空）。  
3. **推荐**：成功时在 **stdout 打印一行 JSON** 或写入 **`--out-json path`**，便于适配器解析；避免不可解析的纯文本混杂。

### 2.2 输入输出与格式

1. **输入**：大文件**一律路径传入**，禁止要求适配器将 GB 级内容塞进环境变量。  
2. **输出**：**必须**产出团队约定的标准制品，例如：  
   - 表格：`CSV`（带表头，UTF-8）；  
   - 报告：`HTML`（可离线打开）；  
   - 归档：`ZIP`（内含上述文件）。  
3. **路径**：**禁止**在算子源码中写死诸如 `/home/alice/...`、`D:\` 等绝对路径；一律通过 **CLI 参数** 或 **环境变量**（由运维注入）指定工作目录与模型目录。  
4. **可复现**：在 README 中注明依赖版本（Conda `environment.yml` 或 `pip-tools` 锁文件）。

### 2.3 安全与资源

1. **禁止**在算子内发起任意出站网络请求（除非经安全评审的固定白名单）。  
2. **禁止** `eval` / `exec` 用户输入。  
3. GPU 使用文档中注明**峰值显存**与建议并发度（默认 **单任务独占**）。

### 2.4 与「适配器」的边界

算子**不得**依赖 `gibh_agent` 包；适配器**不得**把 FastAPI `Request` 对象传入算子。双方契约仅为：**参数 / 文件路径 / 退出码 / stdout/stderr / 输出文件**。

## 3. 智能体适配器开发 SOP（填空题）

### 3.1 注册与元数据

1. 在 `gibh_agent/tools/` 下新增模块（或扩展现有模块）。  
2. 使用 **`from gibh_agent.core.tool_registry import registry`**。  
3. 在异步函数上使用 **`@registry.register(name="...", description="...", category="...", output_type="json")`**。  
   - `name`：**全小写蛇形**或与现有工具保持一致，且与 `[Skill_Route: name]` **逐字一致**。  
   - `description`：供 Tool-RAG / Planner 检索，**写清何时调用、输入类型、关键约束**。  
   - 函数签名中的**可调配参数**应有**类型注解与默认值**，以便自动生成 Pydantic `args_schema`（见 `tool_registry.py` 注释）。

### 3.2 防爆装甲：`@safe_tool_execution`

**当前仓库若尚未在 `gibh_agent/core/utils.py` 中实现该装饰器，必须由架构组先合并等价实现。**

适配器工具函数**必须**满足：

- 任意未捕获异常**不得**冒泡导致 FastAPI Worker 崩溃；  
- 统一转换为 **`{"status": "error", "message": "..."}`**。

**标准用法（示意）**：

```python
from gibh_agent.core.tool_registry import registry
from gibh_agent.core.utils import safe_tool_execution  # 须存在于 utils.py

@registry.register(
    name="example_bio_tool",
    description="一句话说明何时调用。",
    category="Genomics",
    output_type="json",
)
@safe_tool_execution
async def example_bio_tool(file_path: str, threads: int = 4) -> dict:
    ...
```

若暂时无装饰器，**临时**允许在函数体内最外层 `try/except`，但必须在评审单中登记技术债，**限期**改为装饰器统一策略。

### 3.2a 双模输入：文件路径 **或** 内联正文（推荐约定）

技能演示与用户习惯常是「对话里贴序列 / SMILES」，而非先落盘上传。仅要求 `file_path` 会导致模型**臆造** `/app/uploads/demo_xxx.fa` 等不存在的路径，快车道必然失败。

**约定（适配器层实现，算子仍只认路径）**：

1. **工具签名**：除 `file_path: str = ""` 外，增加与场景匹配的可选内联字段（如 `fasta_content`、`smiles_text`、`pdb_content`、`table_content`），**至少二者填一**（有真实上传路径则优先路径）。  
2. **落地路径**：内联正文由适配器写入 **`UPLOAD_DIR` 下专用子目录**（如 `.pyskills_scratch/`），文件名带 UUID，**禁止**与用户示例文件名混用；该目录须与独立算子容器 **挂载同一宿主机卷**（与当前 `worker-pyskills` + `api-server` 共享 `./data/uploads` 一致）。  
3. **技能模板 / 填参提示**：明确写清「可不上传，在 JSON 中填 `fasta_content`，换行用 `\\n`」；并强调 **禁止编造未出现在上传列表中的绝对路径**。  
4. **SkillAgent**：流式/函数填参的系统提示中同步上述规则；无 LLM 回退时可从模板正文 **正则提取** 演示块填入内联字段（兜底）。  
5. **评审清单**：新增检查项——该技能是否在无上传时仍有合法执行路径。

### 3.3 隔离执行：子进程优先，线程池次之

| 场景 | 推荐 API | 说明 |
|------|-----------|------|
| **重计算**（BLAST、GATK、深度学习推理、大内存 R 脚本） | `asyncio.create_subprocess_exec` | 独立进程；可限制 `CUDA_VISIBLE_DEVICES`；崩溃不影响解释器 |
| **阻塞型 C 扩展 / 无子进程 CLI** | `asyncio.to_thread` | 在线程池运行同步函数，**避免阻塞**事件循环 |
| **轻量纯 Python**（毫秒级） | 直接 `await` 协程内逻辑 | 需代码评审确认无阻塞 I/O |

**严禁**：在异步工具函数中直接调用长时间占用的同步 `subprocess.run`、同步 `requests`、或 GPU 推理而不隔离——会导致 **FastAPI 主事件循环饥饿**，全站 SSE 与 API 延迟恶化。

### 3.4 标准工具函数模板（填空题）

将下列骨架复制为起点，替换 `TOOL_NAME`、参数、命令行与解析逻辑：

```python
from __future__ import annotations

import asyncio
import json
import logging
import os
from typing import Any, Dict

from gibh_agent.core.tool_registry import registry
from gibh_agent.core.utils import safe_tool_execution

logger = logging.getLogger(__name__)

# TODO: 由运维配置，禁止写死个人目录
DEFAULT_TIMEOUT_SEC = float(os.getenv("EXAMPLE_TOOL_TIMEOUT", "600"))


@registry.register(
    name="TOOL_NAME",
    description="TODO: 供模型检索的自然语言描述（何时调用、输入是什么）。",
    category="TODO: e.g. Proteomics",
    output_type="json",
)
@safe_tool_execution
async def TOOL_NAME(
    file_path: str,
    some_flag: bool = False,
) -> Dict[str, Any]:
    """
    TODO: 简述工具行为。重计算在子进程中执行。
    """
    py_exe = os.getenv("EXAMPLE_TOOL_PYTHON", "").strip()  # 可选：独立 Conda 解释器
    runner = os.getenv("EXAMPLE_TOOL_RUNNER", "").strip()  # 必填：算子入口脚本

    if not os.path.isfile(file_path):
        return {"status": "error", "message": f"文件不存在: {file_path}"}
    if not runner or not os.path.isfile(runner):
        return {"status": "error", "message": "未配置 EXAMPLE_TOOL_RUNNER 或脚本不存在"}

    cmd = [py_exe or "python3", runner, "--input", file_path]
    if some_flag:
        cmd.append("--verbose")

    proc = await asyncio.create_subprocess_exec(
        *cmd,
        stdout=asyncio.subprocess.PIPE,
        stderr=asyncio.subprocess.PIPE,
        env={**os.environ, "PYTHONUNBUFFERED": "1"},
    )
    try:
        stdout_b, stderr_b = await asyncio.wait_for(
            proc.communicate(),
            timeout=DEFAULT_TIMEOUT_SEC,
        )
    except asyncio.TimeoutError:
        proc.kill()
        await proc.communicate()
        return {"status": "error", "message": f"子进程超时（>{DEFAULT_TIMEOUT_SEC}s）"}

    stderr_text = (stderr_b or b"").decode("utf-8", errors="replace").strip()
    if stderr_text:
        logger.warning("TOOL_NAME stderr: %s", stderr_text[:4000])

    if proc.returncode != 0:
        return {
            "status": "error",
            "message": f"进程退出码 {proc.returncode}；stderr: {stderr_text[:2000]}",
        }

    # TODO: 从约定好的 out-json 路径或 stdout 解析
    try:
        payload = json.loads((stdout_b or b"").decode("utf-8").strip() or "{}")
    except json.JSONDecodeError:
        payload = {}

    # TODO: 映射为与前端契约一致的字段（html_url / csv_url / zip_url 等）
    return {
        "status": "success",
        "message": payload.get("message") or "执行成功",
        "html_url": payload.get("html_url", ""),
        "csv_url": payload.get("csv_url", ""),
        "zip_url": payload.get("zip_url", ""),
    }
```

新文件必须在 **`gibh_agent/tools/__init__.py`**（或项目统一的 tools 聚合入口）中被 import，否则 **`registry` 不会注册**。

## 4. 技能路由与 UI 呈现契约

### 4.1 `seed_skills.py`：`[Skill_Route: tool_name]` 暗号

在 `gibh_agent/db/seed_skills.py` 中，对应技能的 **`prompt_template` 首行或醒目位置**必须包含：

```text
[Skill_Route: tool_name]
```

其中 **`tool_name` 与 `@registry.register(name="tool_name")` 完全一致**（正则仅允许 `[a-zA-Z0-9_]+`）。

**作用链**：

1. 用户从技能广场选择模板 → 会话中带该暗号；  
2. `Orchestrator` 正则命中 → 构造 `SkillAgent`；  
3. `SkillAgent` 使用 `strip_skill_route_tag` 剥离暗号后再送 LLM 填参；  
4. `registry.get_tool(tool_name)` 执行已注册协程。

**用户可读文案（建议沿用）**：`prompt_template` 主体用面向使用者的自然表达（如「你好，我想…」「你可以…」）；需要约束模型如何填 `file_path` / 内联字段时，将字段名与约定**收敛到段末一句**「（助手侧：…）」即可。正文避免对终端用户使用「禁止」「勿删」「兜底」「MVP」「think/JSON」等研发口吻。  
**例外**：七大组学专家角色模板中的「严禁捏造结论」等属于**科研诚信与安全护栏**，与广场演示类技能区分开，可保留。

**常见错误**：

- `tool_name` 与注册名大小写不一致；  
- 技能模板缺少暗号导致走 Planner 长链路，时延与行为不一致；  
- 暗号中含连字符（`-`），正则不匹配。

### 4.2 工具返回字典与呈现（聊天极简 + 右栏万能渲染）

`SkillAgent`（`gibh_agent/agents/skill_agent.py`）对工具返回值做统一处理，**契约如下**：

#### 4.2.1 错误路径

若 `result.get("status") == "error"`：

- 聊天气泡仍使用 **`_format_result_markdown`** 输出「技能执行未完成」与 **`message`**（或 `error`）等详情；  
- **必须**保证 `message` 为**用户可读中文**（或中英可控），禁止仅抛原始堆栈给 UI。

#### 4.2.2 成功路径（链接与数据进右栏，聊天仅一行提示）

若 **非** `error`（含 `status == "success"` 或未显式标错误）：

- **聊天区**：仅推送一句极简 Markdown（例如「技能执行成功，请在右侧工作台查看详细报告」），**不再**在气泡内堆砌 `html_url` / 表格 / JSON。  
- **SSE**：在 `skill_tool_result` 事件与最终 **`done`** 事件的 JSON 中携带 **`tool_name`** 与 **`tool_result`**（完整字典，经 `sanitize_for_json` 可序列化）。  
- **右栏工作台**：由前端根据 `tool_result` 渲染（见 **§4.4**）：有 **`html_url`** 则内嵌 iframe；有 **`data`** 或顶层标量则数据卡片；**`csv_url` / `zip_url` / `json_url`** 映射为下载按钮。

适配器**应优先**将可点击 URL 填入 **`html_url` / `csv_url` / `zip_url` / `json_url`**（支持顶层或嵌套在 `data` 下，与 `SkillAgent._pick_result_url` 一致），以便右栏按钮与 iframe **无缝对接**。

#### 4.2.3 特殊字段

- **`generated_sequence`**：成功时由右栏数据卡片或 `tool_result` 内展示（长序列注意前端截断策略）；**错误路径**下仍可由 `_format_result_markdown` 在聊天区详述。  
- **其它键**：成功时进入 **`tool_result`** 供右栏 JSON 化展示或卡片；**避免**把 megabase 级字符串放在非约定键中。

#### 4.2.4 非 success / 非 error

非 `error` 一律走 **§4.2.2** 右栏呈现；**不推荐**依赖模糊 `status`，新项目请统一 **`success` / `error`**。

### 4.3 执行过程可视化（可选）

快车道执行期间，`SkillAgent` 可通过 SSE 发出 **`[AgenticLog] {...}`**（见 `_agentic_emit`），前端 `handleAgenticLog` 解析为步骤树。**算子层不得依赖该协议**；仅适配器在必要时扩展。

### 4.4 右栏工作台：组学工作流 vs 技能快车道（统一容器、隔离 DOM）

右栏对应 `#workspace-pane` → `.workspace-scroll` → **`#active-workspace-content`**。组学任务与技能结果**共用该根节点**，但**禁止互相覆盖 innerHTML**。

#### 4.4.1 组学分析任务（数据诊断 / 执行步骤 / 专家报告）

- **入口函数**（`services/nginx/html/index.html`）：  
  - **`renderReportSection(section, content, fullData)`**：`section` 为 `diagnosis`（数据诊断报告）或 `summary`（分析结题报告）；在 **`ensureExecutionInstanceContainer()`** 返回的本次执行容器或 `active-workspace-content` 上查找/创建 **`.unified-report-container`**，写入 **`.report-diagnosis-content`** / **`.report-summary-content`**。  
  - **`renderExecutionSteps(stepsDetails)`**：写入同一卡片内的 **`.report-execution-content`**（步骤级 accordion / checklist）。  
- **SSE 来源**：`diagnosis`、`step_result`、`result` 等事件，经 `handleServerEvent` 分发。  
- **占位文案**：`main.css` 中当 **`#active-workspace-content .unified-report-container`** 存在时，隐藏 **`#workspace-placeholder`**。

#### 4.4.2 技能快车道（万能渲染器）

- **挂载点**：在 `#active-workspace-content` 内插入/更新 **`#skill-tool-workspace-root`**（内含 **`.skill-workspace-panel`**），通常 **`insertBefore` 到第一个子节点**，与下方组学 **`.execution-instance` / `.unified-report-container`** **并列**，**不得**修改组学卡片内部结构。  
- **SSE**：  
  - **`skill_tool_result`**：`{ tool_name, tool_result }`，成功执行工具后立即下发，前端 **`case 'skill_tool_result'`** 调用 **`renderSkillToolWorkspace`**，并触发 **`AgentOrchestrator._emit_sse`** 将同一数据写入 **`state_snapshot.skill_last_tool`**（供历史恢复）。  
  - **`done`**：仍携带 **`tool_result`** / **`tool_name`**（与 `consume_pending_done_tool()` 合并），**`handleDoneEvent`** 再次调用渲染（幂等覆盖同一块面板）。  
- **渲染规则摘要**：  
  - 有 **`html_url`**（含嵌套 `data`）→ 全宽高 **iframe**（无 URL 则不建 iframe，避免空白框）；  
  - **`data` 内键值对 + 顶层标量**（排除 URL/status/message 等）→ **数据卡片**；  
  - **仅 CSV/ZIP/JSON 链接** → 仅展示操作按钮行，不强制「无内容」误导文案。  
- **占位文案**：`main.css` 增加 **`#workspace-pane:has(#skill-tool-workspace-root .skill-workspace-panel)`** 时隐藏 **`#workspace-placeholder`**（与组学 `:has(.unified-report-container)` 并列）。  
- **历史会话**：入库的 **`state_snapshot`** 含 **`skill_last_tool`** 时，`createRestoreAIMessageRowWithSnapshot` 恢复气泡后调用 **`renderSkillToolWorkspace`**，无需先跑过 `sendMessage` 闭包。

#### 4.4.3 扩展新技能时最小 checklist

1. 工具返回字典满足 **§4.2.2**（`status`、`message`、链接键、`data` 等）。  
2. 无需改右栏逻辑即可出现按钮/iframe/卡片；若新增展示类型（如专用谱图组件），优先在 **`#skill-tool-workspace-root`** 内扩展子区域，**不要**往 `.report-diagnosis-content` 写入技能专有 DOM。  
3. 样式追加在 **`services/nginx/html/css/style.css`**（`.skill-*` 前缀），避免污染组学 Bootstrap 卡片类名。

## 5.5 交付与运维说明：可执行指令纪律（长记忆 / 协作复用）

人机协作与交接场景中，**只写「该做什么」而不写「怎么执行」**会显著增加排障与轮班成本。本节与 Cursor 工作区规则 **`.cursor/rules/delivery-notes-format.mdc` §3** 对齐，作为后续 Agent 交付与人类 review 的**可复用纪律**（与 §1～§4 的架构条款并列：前者约束代码形态，本节约束**交接可操作性**）。

### 5.5.1 必须满足的写法

1. **凡首次出现、或相对上次交付有变化的运维动作**，在交付说明（精炼总结末尾的「运维与操作说明」）中**必须给出完整、可直接粘贴到终端执行的命令行**（含建议的当前工作目录，例如「在克隆仓库根目录 `/path/to/GIBH-AGENT-V2` 下执行」）。  
2. **禁止**单独使用「请重启相关容器」「更新 worker 即可」等无**对象、无动词参数**的收尾句；至少附一条**可复制的** `docker compose …` / `docker …` / `kubectl …` / `systemctl …` 等全句。  
3. 若服务名、栈文件名、命名空间因环境而异：先给出**仓库默认约定下的一条完整示例命令**，再用括号注明「请将 `worker-pyskills` 替换为你方 compose 中的服务名」等**替换规则**，仍保证读者有一条可运行的基准命令。  
4. **确无 CLI**（例如仅静态资源且热加载、或纯文档变更）：明确写 **「无需在终端执行命令」**，并仍可写一句需留意的配置或缓存路径。

### 5.5.2 示例形态（仅作格式示范，路径请随环境调整）

以下为**格式**示例，实际交付时应替换为与本需求**真实相关**的服务名与 compose 文件位置：

```bash
cd /path/to/GIBH-AGENT-V2
docker compose ps
docker compose build worker-pyskills
docker compose up -d worker-pyskills
docker compose logs -f --tail=80 worker-pyskills
```

Kubernetes 场景则对应给出完整的 `kubectl -n <namespace> rollout restart deployment/<name>` 等（占位符在同一行写清）。

### 5.5.3 与其它文档的关系

- **流式 / 磐石工具协议**：仍以 `docs/METHODOLOGY_Streaming_Thinking_And_Panshi_Tools.md` 为准。  
- **交付分节格式**：以 `.cursor/rules/delivery-notes-format.mdc` 为准；本节为其在「方法论」侧的锚点，便于检索「长记忆」「运维」「可复制命令」等关键词。

## 7. 实践纪要：蛋白表位预测类算子（贯穿全文的一次端到端）

以下用「公开论文配套实现 → 本系统可调用形态」的主线，把 **§1～§5** 串成一条，**不涉及具体仓库目录、函数标识与配置键名**；落地细节见各算法专项运维说明。

**（1）从能力清单到可执行名字**  
能力矩阵里对用户展示的是**可读技能名**；真正触发快车道与工具绑定的，是模板中的 **`[Skill_Route: …]`** 与工具注册名**逐字一致**（见 §1.2、§4.1）。二者不可混用、不可臆造。

**（2）算子侧：先拿到「黑盒」再谈接入**  
在作者公开的版本控制系统中取得源码与入口脚本，在**独立运行时**（虚拟环境或容器）安装依赖，用**命令行**指定输入序列文件与输出目录完成一次完整运行。模型权重必须与该实现**训练与推理假设一致**（例如配套的小型蛋白语言模型骨干）；**更大、不同维度的骨干不能当作即插即用替换**。产出物通常为表格、静态或交互式 HTML、以及按约定编码的序列结果文件等（对齐 §2.2）。

**（3）适配器侧：只通过契约握手**  
编排层**不得**在主进程内长期占用加速器执行上述推理。适配器要么 **fork 出子进程/独立解释器** 调用 CLI（§3.3），要么调用**与本系统解耦的 HTTP 推理网关**，统一解析退出码或响应体，并转换为 **`success` / `error` + 人类可读说明**；成功时把可下载或可打开的地址填入 **HTML / CSV / ZIP** 三类链接约定之一（§3.4、§4.2）。

**（4）数据路径**  
带文件上传时，序列输入应来自 **§1.1** 资产总线解析后的合法路径，或由模型在参数中填入经校验的 FASTA 文本；适配器不负责猜测磁盘布局。

**（5）前端**  
用户从广场选模板 → 会话携带路由暗号 → 快车道进入技能执行链路 → 工具返回经 **§4.2** 格式化的字典后：**错误**仍在聊天气泡详述；**成功**仅气泡提示一行，**详细链接与图表在右栏 §4.4.2 万能渲染器**中展示（`skill_tool_result` + `done` 双通道）。

**（6）最小可核对样例（语义层面）**  
- **输入**：标准 FASTA，一条蛋白序列（含条目行与氨基酸串）。  
- **输出**：同条目下的一条预测序列串，作者文档约定用**大小写**区分「预测为表位相关」与「非表位」的残基位置。  

该样例用于验收「算子单独可跑通 → 适配器能转写为 §4.2 成功形态 → 前端可见链接或等价反馈」，不替代 §5 清单逐项勾选。
